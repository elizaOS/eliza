eliza-1 Model Guidance Review

Generated May 14, 2026. Scope: response handler, planner, evaluator, action schema conversion, validation, cloud/local model adapters, and relevant tests in the elizaOS repository.

Response handler Mostly guided

Schema, grammar, skeleton, span sampler, and field dispatch are wired.

Planner Partially guided

Strict action grammar and validation are wired; strict span argmax and cloud grammar parity are gaps.

Evaluator Under-guided

Uses schema/text prompts, but no skeleton, grammar, span sampler, or guided decode options.

JSON repair Not done

No robust escaping/repair path for raw control characters in JSON strings.

Executive Assessment

The repository has a real guidance layer, but it is unevenly applied. Stage 1 response handling is the most mature: it builds a field-registry schema, response skeleton, GBNF grammar, and span sampler plan, then dispatches through typed field evaluators. Stage 2 planning has a strong strict grammar for action selection and parameters, plus post-hoc validation before action execution. Evaluators are the weak spot: they rely on prompt text and a JSON schema only.

The specific guidance goals are only partially satisfied. Legal-token filtering is implemented through GBNF for response-handler enums and strict planner action/parameter schemas. Argmax sampling is implemented for skeleton enum/number/boolean spans, but the strict planner path currently exposes only a single free-json skeleton span, so action selection is grammar-filtered but not span-sampler argmaxed. Required parameters are structurally emitted by strict planner grammar to a bounded nested depth, then validated before execution. Raw newline/tab/backslash repair inside JSON strings is not implemented; in fact, the shared JSON string GBNF is permissive enough to be risky for raw control characters.

Bottom line: keep the current grammar/skeleton architecture, but close the JSON string escaping gap, bring evaluator onto the same guidance layer, make strict planner span guidance observable, and add end-to-end adapter tests that prove eliza-1 receives the guidance payloads it is expected to honor.

Concrete Examples Found

Response Handler Fields

  • shouldRespond: enum RESPOND, IGNORE, STOP. Grammar filters values; span sampler pins deterministic choice; parser defaults malformed values to RESPOND.
  • contexts: array of allowed context IDs. Grammar pins simple, general, and active context IDs.
  • emotion: enum none, happy, sad, angry, nervous, calm, excited, whisper.
  • relationships: array of objects with required subject, predicate, and object.
  • candidateActionNames: free string array guided by prompt text and later routing, not a closed enum.

Planner and Actions

  • Every exposed action becomes a strict native tool with normalized parameters and compressed routing hints.
  • Strict grammar creates one branch per action: the action literal and its parameter object are coupled.
  • String enum parameters become literal alternatives in grammar and are validated again before execution.
  • Nested object parameters emit required keys first, optional keys afterward, to a finite recursion limit.
  • Terminal actions REPLY, IGNORE, and STOP are always available as core planner tools.

eliza-1 Span Sampler Header Shape

{
  "overrides": [
    { "span_index": 2, "temperature": 0, "top_k": 1 },
    { "span_index": 10, "temperature": 0, "top_k": 1 }
  ],
  "strict": true
}

The header is emitted only for recognized eliza-1 model prefixes. It depends on skeleton spans, so strict planner currently loses most of this benefit because its skeleton is intentionally minimal.

How Each Path Works

Response Handler Flow

  1. The message service builds active response contexts, available action metadata, and a field registry grammar.
  2. It calls runtime.useModel(ModelType.RESPONSE_HANDLER) with native HANDLE_RESPONSE, toolChoice: "required", streamStructured: true, responseSkeleton, grammar, spanSamplerPlan, and guided decode provider options.
  3. Native tool arguments are extracted and normalized into registry fields.
  4. The field registry dispatches handlers, including preemptive handlers for shouldRespond.
  5. Legacy response-handler evaluators may patch contexts, candidate actions, facts, relationships, and reply behavior.

Planner Flow

  1. Actions are tiered, converted to tools, and given strict JSON parameter schemas.
  2. If tools are available, the planner is forced to use native tool calls.
  3. A strict per-action GBNF grammar is attached for local guided decode. Cloud receives native tools but not the grammar body.
  4. Planner output is parsed from native tool calls or fallback JSON, then each call is executed only after argument normalization, enum short-form handling when enabled, schema validation, and action gating.

Evaluator Flow

  1. After a tool result, runEvaluator asks whether to finish, continue, or prefer the next recommended tool.
  2. The call uses responseSchema: evaluatorSchema and text instructions.
  3. Parser rejects mixed JSON/text, normalizes bad decisions to CONTINUE, infers success from terminal tool results in a narrow case, and sanitizes the final user message.

Action Flow

  1. Action metadata is translated to JSON Schema, including enums from enumValues, enum, options, or parameter options.
  2. The planner grammar consumes the supported subset of that schema.
  3. At execution time, validateToolArgs is the hard gate. Invalid calls become structured failures instead of reaching handlers.

Gap Report

Priority Gap Why it matters Recommended fix
P0 Free-string JSON escaping is not enforced or repaired. Raw LF, CR, CRLF, tab, and dangling backslash inside a generated string can make the entire JSON object invalid. Tighten the JSON string grammar to reject raw control characters and add a streaming/local repair layer that converts logical in-string line breaks and tabs to escaped sequences. Add tests for LF, CRLF, CR, multiple blank lines, tab, backslash before quote, and dangling backslash.
P0 Evaluator is not on guided decode. The evaluator decides whether the loop finishes, continues, or asks for another tool. A malformed decision can cause unnecessary loops or premature finish behavior. Add an evaluator skeleton/GBNF builder with deterministic enum/boolean spans for decision and success, then attach withGuidedDecodeProviderOptions in runEvaluator.
P0 Strict planner action selection is grammar-filtered but not span-sampler argmaxed. The code comments expect action enum spans to be pinned, but strict planner returns a single free-json skeleton span, so no action-choice override is emitted. Derive sampler override points from strict grammar branches, add a planner-specific sampler plan, or switch to a skeleton that exposes action and enum parameter spans while preserving strict branch coupling.
P1 Bounded integer grammar appears to emit quoted numeric literals. A schema integer range like 0..5 should generate 0, not "0". If quoted, local decode can create values that fail validation. Change bounded integer literal emission to unquoted numeric GBNF literals and add a planner execution test for bounded integer params.
P1 Validation/reroll utility is implemented but not wired into production source paths. Remote/cloud paths do not receive grammar and can still produce schema-invalid output. The utility is the existing mechanism to retry on remote schema failure. Integrate callModelWithValidation or equivalent schema retry into remote evaluator/planner fallback paths, with local grammar paths explicitly opting out.
P1 Adapter parity is incomplete. Types document grammar, responseSkeleton, and spanSamplerPlan, but inspected cloud and mobile adapters do not forward all of them. Add adapter-level integration tests that assert eliza-1/local requests carry grammar, skeleton metadata, and span sampler controls where supported. Extend adapter request types where needed.
P1 Optional object keys can repeat in grammar. JSON parsers collapse duplicate keys, so validation may not detect that the model generated duplicate optional fields. Emit bounded permutations or a duplicate-safe optional property grammar for small object schemas. For larger schemas, consider post-parse duplicate detection before JSON.parse.
P1 Grammar supports less schema than validation. oneOf, anyOf, patterns, richer numeric bounds, and deep nested objects can fall back to broad JSON, increasing failure probability. Expand grammar support for common schema constructs used by real actions, then add an action-schema fixture suite that compares schema intent to grammar constraints.
P2 Legacy HANDLE_RESPONSE_SCHEMA remains beside field-registry schema. Duplicate schema definitions drift and make audits harder. Consolidate the legacy schema around ResponseHandlerFieldRegistry.composeSchema or remove it once no compatibility path needs it.
P2 candidateActionNames is prompt-guided, not enum-guided. Misspelled action candidates can reduce planner routing quality before the native tool planner stage. Either keep it intentionally fuzzy but canonicalize to known action names, or encode available action names as a legal enum array when the list is small enough.
P2 Enum short-form expansion is off by default. It can recover simple single-enum action args, but changing defaults may hide upstream generation issues. Measure failure recovery from ELIZA_SHORT_FORM_ENUMS=1. If it is beneficial, enable for selected actions or log recoveries before defaulting it on.

Recommended Implementation Order

  1. Fix JSON string safety first: grammar restriction, escape/repair logic where generation is streamed or assembled, and exhaustive newline/tab/backslash tests.
  2. Add evaluator guided decode: schema-to-skeleton/GBNF for decision, success, and optional message fields, plus span sampler pinning.
  3. Repair strict planner span sampler behavior so action and enum params are both legal-token-filtered and argmaxed where eliza-1 supports span overrides.
  4. Fix bounded numeric grammar and add execution-level tests that prove generated params pass validateToolArgs.
  5. Wire remote schema validation/reroll for unguided cloud paths and add adapter parity tests for eliza-1 guidance payloads.
  6. Expand schema coverage in planner grammar based on real action inventory: prioritize enum arrays, nested required objects, numeric bounds, and common oneOf/anyOf shapes.
