Executive Assessment

The repository now has a more consistent guidance layer across the response handler, planner, evaluator, action schemas, and eliza-1 adapters. Stage 1 response handling remains the most mature path: it builds a field-registry schema, response skeleton, GBNF grammar, and span sampler plan, then dispatches through typed field evaluators. Stage 2 planning now exposes the strict action choice to span-sampler argmax while preserving per-action grammar coupling and post-hoc validation. Evaluator output now uses the same guided-decode contract instead of relying on schema and prompt text alone.

The specific guidance goals are now implemented for the main paths. Legal-token filtering is implemented through GBNF for response-handler enums, evaluator decisions, and strict planner action/parameter schemas. Argmax sampling is applied to skeleton enum/number/boolean spans, including planner action choice and evaluator success/decision. Required parameters are structurally emitted by strict planner grammar to a bounded nested depth, then validated before execution. Raw newline/tab/backslash safety is covered by a stricter JSON string grammar and parser repair for invalid provider JSON strings.

Bottom line: the high-priority gaps from the review have been closed in the codebase. Remaining risk is mostly native/provider support: fork-aware eliza-1 backends can honor the guidance payloads, while stock providers may ignore fork extension fields and fall back to schema validation, parser normalization, and reroll behavior.

Concrete Examples Found

Response Handler Fields

  • shouldRespond: enum RESPOND, IGNORE, STOP. Grammar filters values; span sampler pins deterministic choice; parser defaults malformed values to RESPOND.
  • contexts: array of allowed context IDs. Grammar pins simple, general, and active context IDs.
  • emotion: enum none, happy, sad, angry, nervous, calm, excited, whisper.
  • relationships: array of objects with required subject, predicate, and object.
  • candidateActionNames: free string array guided by prompt text and later routing, not a closed enum.

Planner and Actions

  • Every exposed action becomes a strict native tool with normalized parameters and compressed routing hints.
  • Strict grammar creates one branch per action: the action literal and its parameter object are coupled.
  • String enum parameters become literal alternatives in grammar and are validated again before execution.
  • Nested object parameters emit required keys first, optional keys afterward, to a finite recursion limit.
  • Terminal actions REPLY, IGNORE, and STOP are always available as core planner tools.

eliza-1 Span Sampler Header Shape

{
  "overrides": [
    { "span_index": 2, "temperature": 0, "top_k": 1 },
    { "span_index": 10, "temperature": 0, "top_k": 1 }
  ],
  "strict": true
}

The header is emitted only for recognized eliza-1 model prefixes. It depends on skeleton spans, so strict planner currently loses most of this benefit because its skeleton is intentionally minimal.

How Each Path Works

Response Handler Flow

  1. The message service builds active response contexts, available action metadata, and a field registry grammar.
  2. It calls runtime.useModel(ModelType.RESPONSE_HANDLER) with native HANDLE_RESPONSE, toolChoice: "required", streamStructured: true, responseSkeleton, grammar, spanSamplerPlan, and guided decode provider options.
  3. Native tool arguments are extracted and normalized into registry fields.
  4. The field registry dispatches handlers, including preemptive handlers for shouldRespond.
  5. Legacy response-handler evaluators may patch contexts, candidate actions, facts, relationships, and reply behavior.

Planner Flow

  1. Actions are tiered, converted to tools, and given strict JSON parameter schemas.
  2. If tools are available, the planner is forced to use native tool calls.
  3. A strict per-action GBNF grammar is attached for local guided decode. Cloud receives native tools but not the grammar body.
  4. Planner output is parsed from native tool calls or fallback JSON, then each call is executed only after argument normalization, enum short-form handling when enabled, schema validation, and action gating.

Evaluator Flow

  1. After a tool result, runEvaluator asks whether to finish, continue, or prefer the next recommended tool.
  2. The call uses responseSchema: evaluatorSchema and text instructions.
  3. Parser rejects mixed JSON/text, normalizes bad decisions to CONTINUE, infers success from terminal tool results in a narrow case, and sanitizes the final user message.

Action Flow

  1. Action metadata is translated to JSON Schema, including enums from enumValues, enum, options, or parameter options.
  2. The planner grammar consumes the supported subset of that schema.
  3. At execution time, validateToolArgs is the hard gate. Invalid calls become structured failures instead of reaching handlers.

Residual Risk

Priority Original gap Why it matters Recommended fix
P0 Free-string JSON escaping was not enforced or repaired. Raw LF, CR, CRLF, tab, and dangling backslash inside a generated string can make the entire JSON object invalid. Closed Shared grammar now rejects raw controls and invalid escapes; parser repair covers raw line breaks, tabs, controls, and invalid backslashes.
P0 Evaluator was not on guided decode. The evaluator decides whether the loop finishes, continues, or asks for another tool. A malformed decision can cause unnecessary loops or premature finish behavior. Closed Evaluator guidance is now built and attached, with deterministic spans and remote reroll.
P0 Strict planner action selection was grammar-filtered but not span-sampler argmaxed. The code comments expect action enum spans to be pinned, but strict planner returns a single free-json skeleton span, so no action-choice override is emitted. Closed Strict planner now returns a sampler skeleton exposing action choice while preserving strict branch grammar.
P1 Bounded integer grammar appeared to emit quoted numeric literals. A schema integer range like 0..5 should generate 0, not "0". If quoted, local decode can create values that fail validation. Closed Bounded integer literals are emitted as numeric JSON tokens and covered by grammar tests.
P1 Validation/reroll utility was implemented but not wired into production evaluator source paths. Remote/cloud paths do not receive grammar and can still produce schema-invalid output. The utility is the existing mechanism to retry on remote schema failure. Closed for evaluator Evaluator has equivalent remote reroll before parser fallback. Planner native tools remain the primary route; broader remote reroll can still be expanded if text fallback use grows.
P1 Adapter parity was incomplete. Types document grammar, responseSkeleton, and spanSamplerPlan, but inspected cloud and mobile adapters do not forward all of them. Closed Cloud and mobile adapters forward guidance payloads where supported and have focused tests/typechecks.
P1 Optional object keys could repeat in grammar. JSON parsers collapse duplicate keys, so validation may not detect that the model generated duplicate optional fields. Closed Optional keys are emitted at most once in declaration order.
P1 Grammar supports less schema than validation. oneOf, anyOf, patterns, richer numeric bounds, and deep nested objects can fall back to broad JSON, increasing failure probability. Residual Required/nested/enum/numeric common cases are tighter; oneOf, anyOf, patterns, and broad free-form objects remain validation-backed.
P2 Legacy HANDLE_RESPONSE_SCHEMA remains beside field-registry schema. Duplicate schema definitions drift and make audits harder. Residual Left intentionally for legacy compatibility; production field-registry schema remains canonical.
P2 candidateActionNames was prompt-guided, not enum-guided. Misspelled action candidates can reduce planner routing quality before the native tool planner stage. Closed Guidance now prefers known actions, parsed hints are capped, and runtime canonicalization filters to known names/similes.
P2 Enum short-form expansion is off by default. It can recover simple single-enum action args, but changing defaults may hide upstream generation issues. Residual Kept opt-in to avoid masking bad generation; enum schema exposure is improved through deduplication.
