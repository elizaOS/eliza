# Eliza-1 Model Lineup — Windows Validation Research Report

## 1. The Eliza-1 model line (canonical)

The repo's canonical model lineup is defined in three sources of truth that all agree:

- `ELIZA_1_GGUF_PLATFORM_PLAN.json` / `ELIZA_1_GGUF_READINESS.md` (generated by `packages/training/scripts/manifest/eliza1_platform_plan.py`).
- `packages/shared/src/local-inference/catalog.ts` (`ELIZA_1_TIER_IDS`, `FIRST_RUN_DEFAULT_MODEL_ID = "eliza-1-2b"`).
- `plugins/plugin-local-inference/src/services/manifest/eliza-1.manifest.v1.json`.

There are seven text tiers, each shipping as a *bundle* of components: `text/` GGUF, `dflash/` drafter GGUF, `asr/` GGUF (+mmproj), `vad/` GGML, `tts/` (Kokoro ONNX or OmniVoice GGUF), optional `vision/` mmproj, `cache/`, `evals/`, `licenses/`, `quantization/`.

## 3. What "dflash" is

**dflash = Diffusion-Flash speculative decoder**, a custom drafter architecture baked into the elizaOS/llama.cpp fork. Evidence:

- C++ architecture lives at `plugins/plugin-local-inference/native/dflash/dflash_draft.cpp` defining `llama_model_dflash_draft` with GGUF keys `dflash_block_size`, `dflash_mask_token_id`, `dflash_n_target_features`, `dflash_target_layer_ids[]` — it is a **mask-token block-parallel diffusion-style drafter that conditions on a fixed set of target-model hidden features** (consistent with "diffusion-flash speculative" papers, not Gemini-Flash distillation).
- Launched out-of-process by `plugins/plugin-local-inference/src/services/dflash-server.ts`, which spawns the fork's `llama-server` with `-md <drafter.gguf> --spec-type dflash --draft-min --draft-max --n-gpu-layers-draft` and consumes `dflashVerify` / `dflashAccept` SSE events.
- Trained by `packages/training/scripts/distill_dflash_drafter.py`; published to `elizaos/eliza-1/bundles/<tier>/dflash/drafter-<tier>.gguf`.
- Hard-required for every release tier (every tier in `eliza1_platform_plan.py` lists `dflash/drafter-<tier>.gguf` as required, and `LICENSE.dflash` is mandatory).
- AOSP variant: `plugins/plugin-aosp-local-inference/src/aosp-dflash-adapter.ts`.

## 6. Test coverage today

Substantial unit/integration coverage centred on `plugins/plugin-local-inference/src/services/`:
- DFlash: `dflash-server.test.ts`, `dflash-server.p3.test.ts`, `dflash-server-fused.integration.test.ts`, `dflash-structured.test.ts`, `dflash-event-schema.test.ts`, `dflash-verify-event.ts`, `dflash-metrics-collector.test.ts`, `dflash-doctor.test.ts`, `dflash-cache-flow.test.ts`, `__tests__/dflash-checkpoint-client.test.ts`, `manifest/manifest-dflash-eval.test.ts`.
- Voice: `voice/pipeline*.test.ts`, `voice-duet.e2e.test.ts`, `interactive-session.e2e.test.ts`, plus an emotion classifier test.
- Recommendation, ram-budget, gpu-profiles, backend, active-model, catalog, downloader, llama-server-metrics, runtime-dispatcher.
- Stress: `__stress__/cache-thrash.test.ts`, `cache-restart-corruption.test.ts`, `cache-100conv-stress.test.ts`, `cache-multi-model.test.ts`.

These are JS unit/integration tests that should run on Windows under `bun test` / `vitest`. No `describe.skipIf(platform==="win32")` patterns found.

**Hardware-evidence side:** No `evidence/platform/windows-*` files. `PLATFORM_MATRIX.md` confirms every Windows target (`windows-x64-cpu`, `-vulkan`, `-cuda`, `-cuda-fused`, `windows-arm64-cpu`, `windows-arm64-vulkan`) is **"authored-pending-hardware"** — cross-built binaries are not counted, and no real Windows host has produced an evidence JSON. The native runner is ready (`windows_runner.ps1`); nothing has invoked it for the record.

## Prioritized Windows validation checklist

1. **Stage a real `eliza-1-0_8b` bundle on a Windows x64 box** (smallest end-to-end target) — install `~/.eliza/local-inference/models/eliza-1-0_8b.bundle/` with `text/`, `dflash/drafter-0_8b.gguf`, `asr/`, `vad/`, `tts/kokoro/`.
2. **Build the fork for Windows.** `node packages/app-core/scripts/build-llama-cpp-dflash.mjs --target windows-x64-cpu`. Produce `llama-server.exe`, `llama-bench.exe`, `llama-completion.exe`, `CAPABILITIES.json` under `~/.eliza/local-inference/bin/dflash/windows-x64-cpu/`.
3. **Run the canonical Windows runner for CPU**, recording evidence:
   `pwsh -File plugins/plugin-local-inference/native/verify/windows_runner.ps1 -Backend cpu -Model C:\models\eliza-1-smoke.gguf -Report evidence\platform\windows-x64-cpu.json`. The runner expects llama-bench to advertise `tbq3_0`, `tbq4_0`, `tbq3_tcq`, `qjl1_256`, `q4_polar` cache-type aliases.
4. **Repeat for Vulkan** on the same host (requires `vulkaninfo`): `-Backend vulkan` → `evidence/platform/windows-x64-vulkan.json`.
5. **Repeat for CUDA** on an NVIDIA Windows box (requires `nvidia-smi` + `nvcc`): `-Backend cuda` → `evidence/platform/windows-x64-cuda.json`. Required for 4b/9b/27b/27b-256k tiers per readiness.
6. **Run the JS test suite on Windows under `bun test`** for `plugins/plugin-local-inference/src/services/{dflash-*,backend*,downloader,recommendation,manifest,voice}.test.ts`. Capture which (if any) fail on win32 (path separators, child_process spawn, file-locking on cache-bridge tests).
7. **Spawn a live DFlash llama-server** end-to-end via `plugins/plugin-local-inference/src/services/dflash-server.ts` against the 0_8b bundle on Windows: verify `/health`, a `/completion`, and `dflashVerify` SSE events.
8. **Voice path (Kokoro on Windows):** smoke `tts/kokoro/model_q4.onnx` through onnxruntime-node on Windows (no fused-omnivoice on 0_8b/2b/4b — Kokoro-only).
9. **OmniVoice on Windows** (9b+): build `windows-x64-cuda-fused` and validate `libelizainference.dll` + `omnivoice.dll` FFI via `plugin-omnivoice`. Today `authored-pending-hardware`.
10. **Vision/mmproj smoke on Windows** (4b/9b/27b): port `native/verify/eliza1_vision_smoke.mjs` invocation; today only Linux vision-smoke JSONs exist.
11. **ASR bench on Windows:** `tsx native/verify/asr_bench.ts` already branches on `win32` for `libelizainference.dll`. Confirm runs.
12. **windows-arm64 (Snapdragon X / Copilot+):** required for 0_8b/2b only. Cross-build + `windows_runner.ps1 -Backend cpu` then `-Backend vulkan`. Today `authored-pending-hardware`.
13. **Reconcile the `0_6b`/`1_7b` vs `0_8b`/`2b` catalog drift** flagged in `docs/eliza-1-dflash-catalog-fragment.md` before any Windows evidence is treated as canonical.
