## 0. Emotion-handling inventory (what exists today)

**OmniVoice TTS (the base `omnivoice.cpp` we fuse):** *No* emotion category. `src/voice-design.h` has six categories: gender, age, pitch, style (`whisper` only), accent (EN), dialect (ZH) — resolved by `pipeline_tts_resolve_instruct` into a `<|instruct_start|> … <|instruct_end|>` string fed to the Qwen3-0.6B backbone. There is *also* a small set of preserved inline non-verbals (`[laughter]`, `[sigh]`, `[confirmation-en]`, `[question-en]`, …). So base OmniVoice already supports an inline-tag mechanism; it just doesn't have emotion words wired into `voice_design.h`.

**omnivoice-singing (the variant `inference/AGENTS.md` §1 says we ship):** a fine-tuned OmniVoice that adds an emotion+singing tag vocabulary — `[singing]`, `[happy]`, `[sad]`, `[whisper]`, `[angry]`, `[nervous]`, `[calm]`, `[excited]` plus the preserved `[laughter]`/`[sigh]`. Per Wave-6 the legal gate is lifted for non-commercial use; it CAN ship as a default bundle. **Control surface: inline tags in the input text passed to `eliza_inference_tts_synthesize` / `/v1/audio/speech`.** It's a different *GGUF* (the singing fine-tune), not a different ABI — the fused ABI (`ffi.h`, `eliza_inference_tts_synthesize`) is unchanged; the tier manifest's `voice.capabilities` array (`["tts","emotion-tags","singing"]`) tells the runtime whether the bundled TTS GGUF understands the tags. Eval gates exist already: `expressive_tag_faithfulness`, `expressive_mos`, `expressive_tag_leakage` in `eliza1_gates.yaml` (provisional). `Eliza1VoiceSchema` / `manifest/validator.ts` already require an `evals.expressive` block when `capabilities` include emotion-tags/singing. `ensure-local-inference-handler.ts::makeTextToSpeechHandler` deliberately does NOT strip tags ("singing, emotion tags, lyrical phrasing" are passed through).

**Qwen3-ASR (`ggml-org/Qwen3-ASR-{0.6B,1.7B}-GGUF`):** Per upstream (QwenLM/Qwen3-ASR, Jan 2026 release): multilingual speech/music/song recognition, language detection, timestamp prediction, AND **emotion recognition** — surprise, calm, happiness, sadness, disgust, anger, fear (DashScope real-time API exposes "emotion detection" alongside ASR + VAD). The model is reached here via llama.cpp `mtmd` with a `PROJECTOR_TYPE_QWEN3A` mmproj backport (`asr/eliza-1-asr.gguf` + `asr/eliza-1-asr-mmproj.gguf`). **Open question to resolve in impl:** whether the GGUF-converted Qwen3-ASR emits the emotion label as part of the transcript text (a `<emotion>…</emotion>`-style span / a special token) or only via the DashScope server API. The arch doc references `<|instruct_start|>`/`<|instruct_end|>` for ASR prompting and an `<asr_text>`-style wrapping; we need to check `eliza-1-asr-mmproj.gguf`'s special-token map at impl time. If the GGUF doesn't surface emotion, fall back to a tiny emotion-from-audio head (a Silero-sized classifier) OR a heuristic from pitch/energy contour computed in `transcriber.ts` — recorded as "approximated", not faked.

**Runtime today:** ASR detokenize→retokenize round-trip into the text model over `/v1/chat/completions` (per `asr-tokenizer-fusion.md`) — so an emotion span the ASR produces would survive as text through that round-trip; that's the seam to use until Cluster 4's fused token handoff lands.

**Verdict:** emotion through the pipeline is *partially built* — the manifest schema, the eval gates, the TTS tag passthrough, and the omnivoice-singing tag vocab exist; what's missing is (1) the *generation* side (the LLM emitting emotion markup with `replyText`), (2) the *parse* side (extracting it and routing to the TTS), (3) the *perceive* side (reading emotion off Qwen3-ASR), and (4) the *fidelity metric* tying them together.

## A. The two-agents-talking-endlessly harness ("voice:duet")

### A.1 Shape

Two `LocalInferenceEngine` instances in one Bun process (or two, behind a flag), both loading the *same* tier bundle (0.6b first, then 1.7b) but with **different characters** (different `Character` JSON → different `roomId` + system prompt + persona). Agent A's `replyText` → OmniVoice TTS → PCM → fed into agent B's ASR → B's `VoiceTurnController.generate` → B's `replyText` → TTS → A's ASR → … loop forever (or `--turns N`). All tricks on: DFlash spec-decode (`ELIZA_DFLASH_DISABLE` unset), KV-prefix prewarm (`prewarmResponseHandler` via the turn controller's `firePrewarm`), guided structured decode (Cluster 4 — the Stage-1 envelope grammar + the singleton-fill), fused zero-copy streaming (Cluster 4 — `eliza_inference_{tts,asr}_stream_*`). The reduced-optimization fallback is only allowed if a backend genuinely can't dispatch (per `inference/AGENTS.md` §3).

### A.2 Audio routing — in-memory ring hand-off, not speakers/mic

Both agents are local processes, so **never** go through `system-audio-sink.ts` (aplay/afplay) or `mic-source.ts`'s recorder subprocess. Instead:

- Agent A's `VoiceScheduler.sink` is set to an `InMemoryAudioSink` (already exists in `ring-buffer.ts`) whose `write(pcm, sr)` pushes into a `PcmRingBuffer` that is **agent B's `PushMicSource`** (already exists in `mic-source.ts` — "Discord opus-decoded PCM / renderer getUserMedia / the duet harness" feeds it). i.e. `A.scheduler.sink` → ring → `B.micSource.push(pcm)` → B's VAD + streaming transcriber. Symmetrically `B.scheduler.sink` → ring → `A.micSource`.
- The sample-rate seam: TTS emits 24 kHz; the ASR/VAD path wants 16 kHz mono. Resample in the bridge sink (`audio-resample.h` equivalent on the JS side — or feed 24 kHz and let the transcriber's resampler handle it; check `transcriber.ts`'s feed contract). Keep it one resample, zero file writes.
- When Cluster 4's fused streaming TTS lands (`ttsSynthesizeStream` emitting PCM chunks as decoded), the bridge sink drains chunks straight into the peer's ring with no buffering — that's the "first-audio-into-B's-ring" checkpoint (B.1).
- A "duet ring" abstraction: `class DuetAudioBridge { aToB: PcmRingBuffer; bToA: PcmRingBuffer; }` wired by the harness; each direction is one ring, sized via `--ring-ms` (a sweep knob, B.4).

### A.3 Construction — extend `voice-interactive.mjs` → a new `voice-duet.mjs`

New file `packages/app-core/scripts/voice-duet.mjs`, `bun run voice:duet` script in `packages/app-core/package.json`. Reuse from `voice-interactive.mjs`: `inspectActiveOptimizations` (run it for *both* engines, fail-closed on missing prereqs), `PLATFORM_MATRIX`, the latency-trace render. New logic:
- `--model eliza-1-0_6b|eliza-1-1_7b` (default 0_6b), `--turns N` (default infinite), `--character-a path --character-b path` (default: two minimal personas baked in — e.g. "curt" and "chatty" — so it runs with no extra files), `--seed-text "<A's opening line>"`, `--report out.json`, `--ring-ms`, plus all the Cluster-4/B-section sweep knobs (`--parallel`, `--draft-max`, `--ctx-size-draft`, `--prewarm-lead-ms`, chunker thresholds).
- Boot: `engineA.load(bundlePath)` + `engineB.load(bundlePath)` (the same GGUF on disk, two contexts — note `inference/AGENTS.md` §4 wants one process per *engine*, two engines is fine; the fused `llama-server` per engine each holds text+TTS+ASR), `startVoice` + `armVoice` on each, build a `VoiceTurnController` per side whose `generate` runs the runtime message handler (the real Stage-1 forced-grammar path) against that side's character/room, then the loop: seed A's transcriber with `--seed-text` (or A speaks first off an injected transcript), and from then on each side's `onTurnComplete(outcome)` → push `outcome.replyText` through that side's TTS into the cross ring, which the peer's VAD picks up as speech-start → … . Endless until `--turns` or Ctrl-C; on each turn, snapshot both tracers' latest trace into the report.
- Honesty: like `voice-interactive.mjs`, if the bundle / fused `libelizainference` / DFlash binary / required kernels are missing, print the checklist and exit non-zero. No silent stub-TTS duet.

### A.4 The automated e2e test

Extend `voice/interactive-session.e2e.test.ts` (or a sibling `voice-duet.e2e.test.ts`) with:
- **Wiring (unconditional, stub backends):** two `LocalInferenceEngine`s + stub TTS + a `DuetAudioBridge` + two `TestTranscriber`-equivalents + two fake `generate`s that echo a fixed reply. Assert: (a) A's TTS PCM lands in B's ring; (b) B's VAD/transcriber sees it → B's `generate` fires → B's reply lands in A's ring; (c) the loop runs `--turns 3` without deadlock; (d) both latency tracers recorded ≥1 turn each; (e) the cross-ring stays bounded (no unbounded growth).
- **Real-output (gated `it.skipIf(!realBackendPresent)`):** the existing `realBackendPresent` probe (catalog kernels advertised + fused build). When it runs (macOS-Metal / linux fused box with both bundles staged), run a real 3-turn duet on `eliza-1-0_6b` and assert >0 PCM crosses each direction, no crash, the report JSON validates. Don't fake a "real" run.

## B. Latency instrumentation + the scientific grind

### B.1 Extend `latency-trace.ts`'s checkpoint set

Today `VOICE_CHECKPOINTS` = `[vad-trigger, vad-speech-start, prewarm-fired, asr-first-partial, asr-final, llm-first-token, llm-first-replytext-char, phrase-1-to-tts, tts-first-audio-chunk, audio-first-played]` with `t0 = vad-trigger`. For the duet we need a *cross-agent* t0 = **the peer's last-utterance-end** (when A stopped emitting PCM into B's ring). Add:
- `peer-utterance-end` — when the producing agent's scheduler drained its last PCM chunk into the cross ring (the duet bridge marks this on B's tracer). This becomes the **headline t0** for the duet: `peer-utterance-end → llm-first-token` is "TTFT-from-last-utterance", the headline number the TODO asks for.
- `audio-first-into-peer-ring` — when the *responding* agent's first TTS PCM chunk lands in the peer's ring (replaces `audio-first-played` in the duet — there are no speakers). Keep `audio-first-played` for the single-agent harness.
- Keep all the intermediate ones; add `replyText-first-emotion-tag` (C) — when the field extractor sees the first emotion-markup token in `replyText` (so we can measure emotion-markup overhead the way `envelopeToReplyTextMs` measures envelope overhead).
- New derived metrics: `ttftFromUtteranceEndMs` (peer-utterance-end → llm-first-token, **headline**), `replyTextFirstCharFromUtteranceEndMs`, `firstTtsPcmFromUtteranceEndMs`, `firstAudioIntoPeerRingFromUtteranceEndMs`, plus the existing per-stage spans. The `BoundedHistogram` already does p50/p90/p99 — extend `LATENCY_DERIVED_KEYS` and the `DERIVED_SPANS` table; the dev endpoint payload (`buildVoiceLatencyDevPayload`) and `voice-latency-report.mjs` pick the new keys up automatically (they iterate the key array).

### B.2 Non-latency metrics over a long run

The tracer is per-turn span only; add a sibling `VoiceRunMetrics` accumulator (or extend the report writer) recording per-turn + aggregated: **DFlash accept-rate** (from `llama-server /metrics` `n_drafted_accepted/n_drafted` — `dflash-server.ts` already exposes `getDflashRuntimeStatus`; `e2e_loop_bench.mjs` already scrapes this), **structured-decode token-savings %** (Cluster 4 — tokens the runtime force-filled ÷ tokens that would have been generated; Cluster 4's `guided_decode_token_bench.mjs` measures this — feed its counter into the report), **tok/s** (decode tokens / decode wall — server `/metrics`), **RSS over the run** (server `VmHWM` via `/proc/<pid>/status` — `e2e_loop_bench.mjs` does this; flag `leakSuspected` if monotone-increasing over N turns). Emit one `reports/` artifact: `packages/inference/reports/porting/<date>/voice-duet-bench-<model>.json` (and a `.md` summary), schema-aligned with the `eliza1_gates.yaml` gate names so `eliza1_gates_collect.mjs` can ingest it.

### B.3 Feed the gates

`eliza1_gates.yaml` already has `first_token_latency_ms` (≤220/200 ms, provisional, `LatencyTracer ttftMs.p50`), `first_audio_latency_ms` (≤450/400 ms), `barge_in_cancel_ms`, `e2e_loop_ok`, `thirty_turn_ok`, `dflash_acceptance`, `dflash_speedup`, the `expressive_*` gates. Changes: (1) point `first_token_latency_ms` at the new `ttftFromUtteranceEndMs.p50` (it's the same physical quantity for the duet — "speech-end → first token" — currently the gate description says "speech-end to first token" but the tracer measured from `vad-trigger`; the duet's `peer-utterance-end` is the *correct* anchor); (2) add a `duet_round_trip_ms` gate (peer-utterance-end → first-audio-into-peer-ring, p50/p90) — provisional thresholds calibrated from the first real 0.6b/1.7b duet run; (3) add `structured_decode_token_savings_pct` (≥ some floor, provisional); (4) flip `expressive_tag_faithfulness`/`expressive_mos`/`expressive_tag_leakage` to be measured by the C-section emotion-fidelity harness (still provisional until a real omnivoice-singing run reproduces them). All new gates land `provisional: true` until a real-hardware run calibrates them — per the yaml's own contract.

### B.4 The scientific grind loop — knobs + how to drive it

Knobs to sweep (each a `voice-duet.mjs` flag, threaded through to the fused `llama-server` args / scheduler config):
- DFlash: `--parallel` (server slots), `--draft-max` / `--draft-min` (spec-decode draft window), `--ctx-size-draft` (drafter context).
- Chunker: phrase-flush thresholds (`, . ! ?` vs N-words; `phrase-chunker.ts`'s `chunkOn` + word count) — smaller chunks → lower first-audio, worse prosody/RTF; sweep.
- Prewarm: `--prewarm-lead-ms` — the turn controller fires prewarm on `speech-start`; sweep whether prewarming on the *peer's* `speech-pause` (i.e. predicting the turn will end) wins more.
- Rings: `--ring-ms` (cross-ring size) — too small → underrun stalls, too big → added latency.
- KV-cache types: TurboQuant Q3 vs Q4 for KV, QJL on K, PolarQuant on V — these are bundle-level (`recipeManifest`), but the harness can pick among co-staged variants if present.
- Kernel choices: backend selection (Metal/CUDA/Vulkan/CPU fused build) — the harness picks the fused build dir; `--backend` already exists in `e2e_loop_bench.mjs`.

Driver: a thin sweep script `packages/inference/verify/voice_duet_sweep.mjs` that runs `voice-duet.mjs --turns 20 --report …` across a grid of these knobs, collects the p50/p90/p99 + accept-rate + tok/s + RSS into one CSV, and emits a `reports/` "before/after" table. The grind is: profile (the tracer's per-stage histogram tells you the dominant stage), tune that stage's knob, re-run, repeat until the round-trip plateaus. Document the winning config per tier in the manifest's `evals` block + a `reports/` artifact. (On CPU today TTS dominates — RTF ~6–10× per `e2e-loop-benchmark.md` — so the grind is mostly waiting on Cluster-2's GPU-fused build + Cluster-4's streaming TTS; on a GPU-fused build the LLM TTFT + prewarm + chunker thresholds become the lever.)

## C. Emotion through the pipeline

### C.1 The emotion-tag schema

Use the **omnivoice-singing inline-tag vocabulary verbatim** as the canonical schema (no SSML, no new format — it's already what the TTS understands): emotion tags `[happy] [sad] [angry] [nervous] [calm] [excited] [whisper]`, the singing tag `[singing]`, preserved non-verbals `[laughter] [sigh]`. Define one TS constant `EXPRESSIVE_TAGS` (extend `ELIZA_1_VOICE_CAPABILITIES`'s neighborhood in `manifest/schema.ts`) listing the legal tags; a `parseExpressiveTags(replyText): { cleanText, segments: [{text, emotion, singing}] }` helper in a new `voice/expressive-tags.ts`. Tags are *inline* (mid-sentence allowed: `"that's [excited] amazing"`), scoped until the next tag or end of phrase — matches how the chunker already splits, so each `Phrase` carries the emotion in scope when it was cut.

### C.2 Generation side — LLM emits emotion markup with `replyText`

- The voice-output prompt (the system/instruction the runtime message handler uses when the turn is voice — `ensure-local-inference-handler.ts` / the response handler) gains a clause: "When the turn is spoken, you MAY annotate `replyText` with inline expressive tags from this set: [happy] [sad] [angry] [nervous] [calm] [excited] [whisper] [singing] [laughter] [sigh]. Use them sparingly and only when the affect is genuine. Tags are inline; they do not appear in text-only turns." Gate on `manifest.voice.capabilities` including `emotion-tags` (don't instruct the model to emit tags a base-TTS bundle can't render → would just leak the literal).
- **The Stage-1 / `replyText` schema may grow an `emotion` field** — instead of (or in addition to) inline tags, the Stage-1 envelope could carry `{ shouldRespond, replyText, contexts, emotion?: "happy"|"sad"|… }` as a top-level enum field. **Cluster 4's structured decode handles this**: the grammar's enum-narrowing → singleton-fill fills the `emotion` value deterministically once the prefix narrows it, costing zero generation tokens. Decision to make with Cluster 4: inline-tags-in-replyText (more expressive, mid-sentence) vs a single envelope `emotion` enum (cleaner, fully structured-decode-able, less expressive) — likely *both*: an envelope `emotion` as the "dominant affect" + optional inline tags for mid-sentence shifts. Either way the parse path (C.3) handles whatever form ships.
- **This is in Cluster 3's fine-tune corpus**: the `eliza_native_v1` trajectory shape gets the expressive tags / `emotion` field populated on voice-mode rows (Cluster 3 augments via Cerebras `gpt-oss-120b` — generate the affect annotation alongside the reply). So the 0.6b fine-tune learns to emit emotion markup natively. Coordinate the exact schema (inline vs envelope field) with Clusters 3 + 4 before the corpus is generated.

### C.3 Parse path — `replyText` → OmniVoice emotion/singing controls

`parseExpressiveTags(replyText)` runs in the `generate` callback (or in `VoiceScheduler` before the chunker) → each `Phrase` handed to the TTS backend carries `{ text, emotion?, singing? }`. The `OmniVoiceBackend.synthesize` args grow `emotion?: string; singing?: boolean` (or the phrase keeps the inline tags and the FFI layer passes the tag-bearing text straight through — simpler, since omnivoice-singing parses the tags itself; the `SpeakerPreset` / `instruct` string is where any *non-inline* emotion would go via `voice_design.h` if we extend it, but the singing variant uses inline tags so just **pass the tags through in the text**). Net: `replyText` (with tags) → chunker → `Phrase.text` (tags preserved within the phrase's span) → `eliza_inference_tts_synthesize(text=phraseWithTags)` → omnivoice-singing renders the affect. The `tagLeakage` eval guards that the literal `[happy]` never reaches the audio. For a *base-TTS* bundle, `parseExpressiveTags` strips the tags before TTS (and the LLM shouldn't have emitted them — but defense in depth).

### C.4 The emotion-fidelity metric

For each duet turn: A intends an emotion (the dominant tag / envelope `emotion` it emitted) → A's TTS renders it → B's ASR perceives an emotion (Qwen3-ASR's emotion output, or the fallback classifier from C.0). **`emotionFidelity` = agreement between A's intended emotion and B's perceived emotion**, over the loop — a confusion matrix + an accuracy/F1. Mapping: Qwen3-ASR's labels (surprise/calm/happiness/sadness/disgust/anger/fear) ↔ the tag vocab (happy/sad/angry/nervous/calm/excited) — define an explicit mapping table (`happy↔happiness`, `sad↔sadness`, `angry↔anger`, `nervous↔fear`, `calm↔calm`, `excited↔surprise`; `whisper`/`singing` are style not emotion → excluded from the fidelity score, scored separately as "style preserved" via the `instruct`-channel round-trip). Report it in the same `reports/` artifact as the latencies, one row per turn + aggregate. This feeds the `expressive_tag_faithfulness` gate (rename/alias to `emotion_fidelity` or keep `tagFaithfulness` = the same number). Honesty: if Qwen3-ASR's GGUF doesn't surface emotion, the metric is computed off the fallback classifier and labeled `"perceiver: fallback-classifier"` in the artifact — recorded, not faked; if neither is available, the metric is `null` ("needs an emotion-aware ASR / classifier"), not a fabricated number.

## D. Cross-cluster dependencies & sequencing

- **Needs Cluster 4 (guided structured decode + fused zero-copy streaming):** the duet's "all tricks on" requires the singleton-fill structured decode (for the envelope + the new `emotion` field) and the real fused streaming ABI (`eliza_inference_{tts,asr}_stream_*`) for the chunk-granularity ring hand-off. Until C4's streaming lands, the duet runs on batch TTS/ASR (slower first-audio, the `e2e_loop_bench.mjs` story) — still a valid harness, just not the final latency story. The `structured_decode_token_savings_pct` metric reads C4's token counter.
- **Needs Cluster 3 (fine-tune corpus):** the 0.6b/1.7b fine-tune must learn the expressive-tag / `emotion`-field schema; coordinate the exact schema (C.2) with C3 *before* corpus generation. The duet's *quality* (on-topic replies) needs the fine-tune — on base-v1 weights the replies are off-topic (LaTeX), per `e2e-loop-benchmark.md`; the loop still exercises the full path.
- **Needs Cluster 2 (kernel parity + the full build matrix):** a GPU-fused `libelizainference` (`linux-x64-{cuda,vulkan}-fused`, `darwin-arm64-metal-fused`) is required for the latency story to be good (CPU MaskGIT TTS is ~6–10× RTF — TTS dominates the round-trip). The grind (B.4) is gated on C2 producing a GPU-fused build on the local RTX 5080 / the H200. The kernel-set completeness (`turbo3_tcq`, `qjl_full`, `polarquant` on the fused build) is a publish blocker C2 owns.
- **Needs Cluster 1 (hygiene):** the new `voice-duet.mjs` / `voice-duet.e2e.test.ts` / `latency-trace.ts` changes / `expressive-tags.ts` must land lint+typecheck-clean; the `conversation-registry` mock-pollution leak C1 is fixing affects whether `bun test packages/app-core/src/services/local-inference` (which the duet test lives under) is green.
- **Feeds back to:** Cluster 3 (the schema), Cluster 4 (the `emotion` field in the Stage-1 schema → its grammar), the manifest `evals` block + `eliza1_gates.yaml` (the new gates), `packages/inference/AGENTS.md` §8 (the duet round-trip is part of "done").

Cluster 3 must build/stage the singing GGUF for the expressive path to be real; until then the emotion metric runs but `tagFaithfulness` will be poor (base TTS ignores the tags).
