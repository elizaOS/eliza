### Cluster 4 — Guided structured decode + fused zero-copy streaming
- **Guided structured decode (the schema-driven deterministic-token-fill):** verify eliza-1 *has* this; if not, implement it: given the eliza-harness response schema (the Stage-1 envelope, action params, evaluator params, enums), when the model is partway through a parameter value and the remaining-valid-tokens set is a singleton (or the param's value can only be one of a known set and the prefix narrows it to one), **the runtime fills in the rest of that value, closes it, fills in `","<next-param-key>": "` deterministically, and only lets the model infer the *next* parameter's value** — generation tokens are spent only where there's real choice. This rides on the existing `buildResponseGrammar`/`compileSkeletonToGbnf`/`resolveGrammarForParams` + `prefill`/`responseSkeleton`/`grammar` plumbing + the fork's `grammar_lazy` — extend it to the "narrow-to-singleton → auto-complete + advance" behavior (a GBNF that the server's grammar engine collapses, or a token-injection loop in `dflash-server.ts` that detects the singleton state from the grammar's allowed-token mask). Issue a sub-agent to audit the existing work on this and the gap. Make it work on the dflash spec-decode path (the drafter should draft the *value* tokens; the forced spans cost zero draft/verify cycles). Wire it into the e2e (Cluster 5) and the fine-tune corpus (Cluster 3).
- **Fused / zero-copy streaming text↔audio↔text:** stream the ASR's token output directly into the LLM (no file, no full-buffer round-trip — frame/token granularity), and the LLM's `replyText` tokens directly into the TTS (token granularity into OmniVoice, or sub-phrase chunks if the codec needs them), with the audio output of one stage drained directly into the next stage's input buffer — minimal copies, zero file writes, zero added latency. Today: the fused `libelizainference` serves text + `/v1/audio/speech` on one PID but the streaming ASR/TTS symbols are honest stubs (`asr_stream_supported()==0`); the real fused streaming decoder ("W7" ABI — `eliza_inference_{asr,tts}_stream_*`) needs implementing; the within-turn `madvise(MADV_DONTNEED)` RSS trim is open. Implement the real streaming decoders in the fused build (OmniVoice incremental synth emitting PCM as decoded; Qwen3-ASR incremental decode emitting partials), wire the ring-buffer hand-off so stage N's output ring is stage N+1's input ring (or a zero-copy `mmap`'d shared region), and — investigate whether the *weights* of adjacent stages can be co-resident in one llama.cpp context / one ggml graph (the "fuse the models directly" idea — e.g. ASR-encoder → text-decoder → TTS-decoder as one graph with shared KV/scratch) so there's literally no inter-process anything. Research first; then implement what's tractable; document what isn't. Issue a sub-agent to research the streaming/fusion design and any existing work.

### Cluster 5 — The two-agents-talking-endlessly latency benchmark + emotion through the pipeline
- **The e2e test (the headline deliverable):** the **smallest model (0.6b, then 1.7b)** runs as agent A; another instance of the *same model* (different *character*/persona) runs as agent B; A speaks → its `replyText` goes through the **OmniVoice TTS backend** → the audio goes through B's **ASR** → B responds → through TTS → through A's ASR → … **endlessly** (with all the tricks on: dflash spec-decode, KV prewarm, guided structured decode from Cluster 4, fused zero-copy streaming from Cluster 4, the reduced-optimization fallback only if a backend can't dispatch). Build it as a runnable harness (`bun run voice:duet` or similar — extend `voice-interactive.mjs`) + an automated e2e test (gated where it needs the real model/voice build).
- **Latency instrumentation (scientific):** measure and report (p50/p90/p99 over many turns) every inter-stage latency — last-utterance-end → VAD-endpoint → ASR-final → first-LLM-token (TTFT-from-last-utterance is THE headline number) → first-`replyText`-char → first-TTS-PCM-chunk → first-audio-played-into-B's-mic; plus the dflash acceptance rate, the structured-decode token-savings %, the tokens/s, the RSS over a long run. Feed these into `eliza1_gates.yaml` + a `reports/` artifact. Then **scientifically grind**: iterate on the bottlenecks (profile, tune `--parallel`/`--draft-max`/`--ctx-size-draft`/the chunker thresholds/the prewarm timing/the ring-buffer sizes, the KV-cache types, the kernel choices) until the round-trip is as fast as it can be on 0.6b then 1.7b. Document the before/after.
- **Emotion through the pipeline (issue a dedicated sub-agent):** evaluate how emotion is parsed / generated / interpreted at each stage — (a) the ASR side: does Qwen3-ASR surface speaker emotion / prosody, and can we tag the transcript with it? (b) the LLM side: should the voice-output prompt instruct the model to emit emotion tags / SSML-ish markup with `replyText`, and should that be in the fine-tune corpus (Cluster 3)? (c) the TTS side: **omnivoice-singing already supports emotion/singing tags** — make sure the emotion the LLM intended is parsed out of `replyText` and passed into OmniVoice's emotion/singing controls. (d) the loop: compare the emotion B's ASR *perceives* in A's speech against the emotion A *intended* — an emotion-fidelity metric across the loop, reported alongside the latency numbers. The sub-agent: research what emotion handling exists in Qwen3-ASR / OmniVoice / omnivoice-singing / the prompts / the training data, design the emotion-tag schema + the parse/generate/interpret path, and feed its design back into Clusters 3 (fine-tune corpus), 4 (the `replyText` schema may grow an `emotion` field/markup → structured-decode handles it), and 5 (the metric).
