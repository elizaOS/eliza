# Eliza-1 Metal kernel perf bench — Apple M4 Max, 2026-05-10

Hardware: Apple M4 Max (40-core GPU), Darwin 25.2.0, 546 GB/s peak unified
memory bandwidth.

Harness: `verify/metal_bench.mm` — runtime-JIT compiles each shader via
`MTLDevice.newLibraryWithSource`, allocates 9B-class production-sized
buffers once, runs 50 interleaved warmup passes, then measures 1000
interleaved iterations. Per-dispatch GPU time is measured via
`MTLCommandBuffer.GPUEndTime - GPUStartTime` (entitlement-free, µs
resolution). CPU time wraps `commit` → `waitUntilCompleted`.

Workload: single decode-step attention scoring at one transformer layer.
`head_dim=128`, `seq_len=4096`, `n_kv_heads=32`. Production extrapolation
assumes a 9B model with **80 layers**.

**Verification:** all 5 shaders re-tested via `metal_verify` after the
bench harness was built — **8/8 PASS** retained on every kernel
(turbo3, turbo4, turbo3_tcq, qjl, polar). Bench is additive; nothing
under `metal/` was touched.

| `turbo3`     | 228.3           | 616.5        | 367             | 34.4 GB/s | 6.3 %         |
| `turbo4`     | 229.4           | 611.8        | 353             | 40.0 GB/s | 7.3 %         |
| `turbo3_tcq` | 229.7           | 616.0        | 350             | 32.0 GB/s | 5.9 %         |
| `qjl`        | 228.5           | 604.7        | 347             |  7.3 GB/s | 1.3 %         |
| `polar`      | **4 215.8**     | **4 565.4**  | 4 345.5         |  2.7 GB/s | 0.5 %         |

`bytes_per_dispatch` for the bandwidth column counts Q/q_sketch input,
the full packed K-cache for one decode step's worth of tokens
(131 072 blocks for turbo*, 32 768 for qjl, 131 072 for polar), and the
fp32 score output buffer.

## Bandwidth-bound vs compute-bound

- **Most bandwidth-bound:** `turbo4` at 40 GB/s (7.3 % of peak), with
  `turbo3` close behind. The turbo* family streams ~7–9 MB of packed K
  per dispatch and lands at the highest realised bandwidth — but is
  still ~14× under the 546 GB/s ceiling, so it is **launch-overhead-
  bound, not memory-bound**. The four ~228 µs kernels share an almost
  identical GPU floor: this is the kernel-launch + 131 072-threadgroup
  scheduling cost on M4 Max for a 32-thread one-SIMD-group dispatch.
- **Most compute-bound:** `polar`, by a factor of **~18×** vs the turbo
  family. The shader explicitly serialises the 128-element Walsh-
  Hadamard butterfly inside `tid==0` (per the `// thread does the work`
  comment in `metal/polar.metal`), so 31 of 32 SIMD lanes idle through
  every butterfly stage. At 0.5 % of peak BW it is clearly not memory-
  starved — it is compute-starved by the serial butterfly. This is the
  single biggest perf optimisation lead in the suite.
- **`qjl`** sits at 1.3 % of peak BW because the QJL block is 7.5×
  smaller than fp16 K (34 B vs 256 B per token), so absolute bandwidth
  demand is small even though the dispatch count is identical. It is
  not bandwidth-limited; it is launch-overhead-limited like turbo*.

## Tokens/sec equivalent (single-kernel-only, 9B model, 80 layers)

If a 9B decode step must run this kernel once per layer, the upper
bound on per-decode throughput from this kernel alone is:

| Kernel       | Tokens/sec (this kernel only) |
|--------------|------------------------------:|
| `turbo3`     | 54.8                          |
| `turbo4`     | 54.5                          |
| `turbo3_tcq` | 54.4                          |
| `qjl`        | 54.7                          |
| `polar`      |  **3.0**                      |

The four launch-overhead-bound kernels collectively cap a 9B decode at
~14 tok/s (sum of latencies = 4×229 µs = 916 µs/layer × 80 layers ≈
73 ms/token) **before** matmul/MLP/RMSNorm/RoPE/sampler are added —
which suggests on real hardware the dispatch geometry should batch
multiple layers' KV ops into one command buffer. `polar`'s 3.0 tok/s
is alarming; if it stays in the per-decode path it dominates total
latency. **Recommended follow-up for shader-review agent: parallelise
the polar Hadamard butterfly across the 32 SIMD lanes.**

## TurboQuant Q4 KV vs fp16 KV — speedup estimate

For one decode step, with 32 KV heads × 4096 tokens × 128 head_dim:

|              | Bytes per token | Bytes per layer-decode | Memory traffic / layer |
|--------------|----------------:|-----------------------:|-----------------------:|
| fp16 K + V   | 512 B           | 67 MB                  | 67 MB                  |
| TurboQuant Q4 K | 66 B          | 8.6 MB                 | 8.6 MB                 |
| TurboQuant Q3 K | 14 B          | 1.8 MB                 | 1.8 MB                 |
| QJL K-cache     | 34 B          | 4.5 MB                 | 4.5 MB                 |
| Polar Q4 V      | 82 B          | 11 MB                  | 11 MB                  |

The TurboQuant Q4 K-cache compresses 7.8× vs fp16 K. At the realised
turbo4 bandwidth (40 GB/s), the kernel reads 8.6 MB in ~215 µs of
GPU time; an fp16-K equivalent at the same realised bandwidth would
need ~1.7 ms — a **~7.8× speedup**. Note this is the speedup on the
KV-read step only, and is bounded by realised, not peak, bandwidth.
On a memory-bandwidth-bound real workload (32k+ context) the speedup
ratio holds; at this 4k context it is dominated by launch overhead
on both sides, so the in-the-wild measured speedup will be smaller
until the dispatch is properly batched.

## Memory savings, K + V cache, 9B model (80 layers, 32 KV heads, head_dim 128)

Per-token storage in MB across the full KV cache:

| Format combo         | Per-token bytes | 8 k context  | 32 k context | 64 k context |
|----------------------|----------------:|-------------:|-------------:|-------------:|
| fp16 K + fp16 V      | 1 310 720       | 10.0 GB      | 40.0 GB      | 80.0 GB      |
| Q4 K + fp16 V        |   823 296       |  6.3 GB      | 25.1 GB      | 50.2 GB      |
| QJL K + Polar Q4 V   |   296 960       |  **2.3 GB**  |  **9.1 GB**  | **18.1 GB**  |
| QJL K + fp16 V       |   742 400       |  5.7 GB      | 22.7 GB      | 45.3 GB      |
| Q4 K + Polar Q4 V    |   379 904       |  2.9 GB      | 11.6 GB      | 23.2 GB      |

(Per-token bytes = 80 layers × 32 kv_heads × per-format-bytes-per-token,
where Q4 K = 66 B, QJL K = 34 B, fp16 K = 256 B, Polar Q4 V = 82 B,
fp16 V = 256 B.)

The headline: **QJL K + Polar Q4 V cuts the 64k-context KV cache from
80 GB (fp16) to 18 GB — a 4.4× reduction, and a 2.8× reduction vs the
Q4-K + fp16-V baseline.** This is what makes 64k context viable on
9b's 24 GB / 48 GB device tiers (per `AGENTS.md` §2).

## What the perf-counter gap looks like

Apple Silicon exposes `MTLCounterSampleBuffer` for stage-boundary
counters (`MTLCommonCounterTimestamp` works without entitlements at
the command-buffer boundary, which is what we use), but the richer
GPU-internal counter set (cycles, ALU active, memory stalls, L2 hits)
requires the developer-mode entitlement and root-equivalent
provisioning. metal_bench documents this in the JSON's
`counter_sample_buffer.note` field; the gap is acknowledged.

## Surprises

1. The four ~228 µs kernels' GPU times cluster within ±0.7 % of each
   other despite turbo3 reading 7.9 MB and qjl reading 1.7 MB. This
   isn't a bandwidth measurement — it's the M4 Max command-encoder +
   threadgroup-launch floor for 131 072 single-SIMD-group dispatches.
   Real production wins from these kernels need fewer, larger
   threadgroups, or batched layer-fused command buffers.
2. `polar` is the outlier and the obvious optimisation lead. The
   per-block serial butterfly is leaving 31/32 lanes idle; a
   threadgroup-cooperative butterfly should reclaim ~10×.
3. `bandwidth_pct_of_peak` is single-digit-% for every kernel — the
   M4 Max's 546 GB/s ceiling is nowhere near the bottleneck at this
   dispatch shape. The bandwidth column is informative for the
   compression-ratio comparison but should not be read as "this kernel
   is bandwidth-limited".
