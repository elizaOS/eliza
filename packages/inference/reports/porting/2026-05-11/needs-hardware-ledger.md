# Eliza-1 Local Inference — "Needs Hardware" Ledger - 2026-05-11

This is the canonical "what hardware does someone need to plug in" doc. It lists
every backend × device that is **source-complete but unverified on real
hardware**, with the exact runner command and the remaining blocker. It is
deliberately narrow: things that already have a hardware run live in
`remaining-work-ledger.md`'s "Current Runtime Truth" table, not here.

Three states (same vocabulary as `remaining-work-ledger.md`):

- **shader-verified** — a standalone shader matches a JSON fixture (possibly on
  real hardware).
- **symbol-shipped** — the patched fork artifact contains the kernel symbol.
- **runtime-ready** — llama.cpp graph execution can select the kernel and a
  built-fork smoke numerically matches the reference, on real hardware.

Only **runtime-ready** satisfies the Eliza-1 publish contract.

Wherever a row says "verified here", it means an actual hardware run on the box
this ledger was written from (Intel Arrow Lake / Mesa ANV 25.2.8 Linux for
Vulkan/CPU; Apple M4 Max for Metal/MoltenVK; iPhone 15 Pro / iOS 26.3.1 for the
iOS device smoke). "needs-hardware" rows have **no** such run.

## Notes / open decisions

- **ROCm fixture harness (row 8) — DONE (2026-05-12).** `packages/inference/verify/hip_verify.cu` now exists: a thin shim that `#include`s `cuda_verify.cu` (which guards its backend headers on `__HIP_PLATFORM_AMD__` and aliases the `cuda*` runtime calls to `hip*`), so `make -C packages/inference/verify hip-verify` runs the exact same ~25 device kernels + fixture loader + reference cross-check the NVIDIA `cuda-verify` does, compiled by `hipcc` — no GGUF needed. Source authored + buildable; no `hipcc` on the authoring box (clean "install ROCm" message). ROCm still has **no HIP port of the production custom kernels** (turbo/qjl/polar/turbo-tcq `.cu` aren't `__HIP_PLATFORM_AMD__`-clean yet) — so a ROCm *runtime* build today is the canonical case for the **reduced-optimization local mode** (`ELIZA_DFLASH_ALLOW_REDUCED_KERNELS=1` at build time, `ELIZA_LOCAL_ALLOW_STOCK_KV=1` at runtime: stock f16 KV, loud warning, `publishable: false`, NOT a default).
- **`turbo3_tcq` as a generic K/V cache type — DONE (2026-05-12).** The `[GGML_TYPE_TBQ3_TCQ] = {...}` type-traits entry now exists in the fork's `ggml.c` (`to_float = dequantize_row_tbq3_tcq` sliding-9-bit-window codebook lookup; `from_float_ref = quantize_row_tbq3_tcq_ref` host-side 512-state Viterbi forward+backtrack in the orthogonal WHT basis with per-block norm re-correction; codebook + the two 128-element WHT sign vectors in the new `ggml/src/ggml-tcq-codebook.h`), plus the matching ggml-cpu `from_float` so `ggml_cpy(K_cur -> K_view)` can quantize, plus the `quantize_tbq3_tcq` size wrapper + the `validate_row_data` case. Decode is bit-identical to the CUDA (`ggml-cuda/turbo-tcq.cu`) + Metal decode kernels. Like `qjl1_256`/`q4_polar`, TCQ is a cache type only (never a `mul_mat` operand); the score path consumes the WHT-rotated cache directly. `build-llama-cpp-dflash.mjs`'s `patchServerKvCacheTypeNames` now appends `GGML_TYPE_TBQ3_TCQ` to `common/arg.cpp`'s `kv_cache_types` (V2 sentinel, tolerates a prior V1 patch), so `--cache-type-k tbq3_tcq` resolves. Fork branch: `elizaOS/llama.cpp` `eliza/ws2-tbq3-tcq-traits` @ `536ff214` (on top of the prior pin `2f80716c`); the `packages/inference/llama.cpp` gitlink is bumped to it. WS-4 to merge the decode-loop / streaming-decoder source on that branch, then tag. Verified: `ggml-base` + `ggml-cpu` compile clean, `quantize_row_tbq3_tcq_ref` / `dequantize_row_tbq3_tcq` / `quantize_tbq3_tcq` symbols exported. This unblocks the 27b / 27b-256k / 27b-1m tiers (`requiredKernelsForContext` adds `turbo3_tcq` at `contextLength >= 65536`). The CPU *standalone score graph op* stays `reference-only` (no public CPU ggml graph builder for the TCQ score in the pin — the reduced-mode hatch covers that).

- **TPU / NPU — not a target this wave (verdict).** The eliza-1 text backbone doesn't fit a Coral Edge TPU's 8 MB SRAM, isn't int8-only quantizable to it, and KV-cache attention isn't an Edge-TPU workload; the Pixel Tensor TPU has no public third-party delegate API; NNAPI is deprecated. The Android GPU (Mali/Adreno via Vulkan) is the on-device accelerator (already targeted). The sidecars don't win enough; OmniVoice TTS is fused into the llama.cpp build (pulling it onto a separate NPU breaks the §4 one-process/one-build fusion contract). The only open angle: a `ELIZA_VAD_QNN_DELEGATE=1` flag that runs Silero VAD on the Hexagon NPU when `onnxruntime-mobile` is built with the QNN EP — a **battery** optimization for always-listening wake-word mode, not latency; stretch, not core. No `plugin-coral` / `plugin-qnn`.
- **Reduced-optimization local mode (the "works everywhere regardless of GPU" hatch).** `backend.ts`'s `localAllowStockKv()` + `warnReducedOptimizationLocalMode()`, `dflash-server.ts`'s `resolveCacheTypeForBackend()`, and `build-llama-cpp-dflash.mjs`'s `writeCapabilities()` (`reducedOptimizationLocalMode` field + the `ELIZA_DFLASH_ALLOW_REDUCED_KERNELS` opt-in). Default behaviour is unchanged (hard refusal); this is opt-in, loud, and non-publishable.
