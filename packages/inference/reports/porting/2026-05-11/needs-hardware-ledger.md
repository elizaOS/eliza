# Eliza-1 Local Inference — "Needs Hardware" Ledger - 2026-05-11

This is the canonical "what hardware does someone need to plug in" doc. It lists
every backend × device that is **source-complete but unverified on real
hardware**, with the exact runner command and the remaining blocker. It is
deliberately narrow: things that already have a hardware run live in
`remaining-work-ledger.md`'s "Current Runtime Truth" table, not here.

Three states (same vocabulary as `remaining-work-ledger.md`):

- **shader-verified** — a standalone shader matches a JSON fixture (possibly on
  real hardware).
- **symbol-shipped** — the patched fork artifact contains the kernel symbol.
- **runtime-ready** — llama.cpp graph execution can select the kernel and a
  built-fork smoke numerically matches the reference, on real hardware.

Only **runtime-ready** satisfies the Eliza-1 publish contract.

Wherever a row says "verified here", it means an actual hardware run on the box
this ledger was written from (Intel Arrow Lake / Mesa ANV 25.2.8 Linux for
Vulkan/CPU; Apple M4 Max for Metal/MoltenVK; iPhone 15 Pro / iOS 26.3.1 for the
iOS device smoke). "needs-hardware" rows have **no** such run.

## Notes / open decisions

- **ROCm fixture harness (row 8).** Recommendation: write `packages/inference/verify/hip_verify.cu` (compiled with `hipcc`) that mirrors `cuda_verify.cu` — same fixture byte images, same reference algorithm ported to HIP `__device__` kernels, linking the same `qjl_polar_ref.o`. That gives ROCm a backend-local numeric gate that does not require a GGUF model. Until then, ROCm's only check is the model-backed `rocm_runner.sh` graph smoke. ROCm also has **no HIP port of the custom kernels** (turbo/qjl/polar `.cu` aren't `#ifdef __HIPCC__`-clean yet) — so a ROCm build today is the canonical case for the **reduced-optimization local mode** (`ELIZA_DFLASH_ALLOW_REDUCED_KERNELS=1` at build time, `MILADY_LOCAL_ALLOW_STOCK_KV=1` at runtime: stock f16 KV, loud warning, `publishable: false`, NOT a default).
- **`turbo3_tcq` as a generic K/V cache type.** `block_tbq3_tcq` has a layout in `ggml-common.h` but **no ggml type-traits entry in `ggml.c`** (no `[GGML_TYPE_TBQ3_TCQ] = {...}` with `to_float`/`from_float_ref`), so `--cache-type-k turbo3_tcq` can't be wired the way `qjl1_256`/`q4_polar` can (those have full type traits and `patchServerKvCacheTypeNames` adds them to the `kv_cache_types` whitelist). TCQ is currently an attention-score-only op. A CPU/Vulkan/CUDA `llama-server` build therefore advertises `{dflash, turbo3, turbo4, qjl_full, polarquant}` but not `turbo3_tcq`; since `turbo3_tcq` is only required for the 27b-256k context tiers (`requiredKernelsForContext` adds it at `contextLength >= 65536`), `eliza-1-1_7b` (32k) loads on those builds without the reduced-mode hatch. To make `turbo3_tcq` a real cache type, add the type-traits entry + a `quantize_row_tbq3_tcq_ref` / `dequantize_row_tbq3_tcq` pair (the Viterbi encoder + the sliding-window decoder from `reference/turbo_kernels.c`) — fork-side work, out of SA-1's scope.
- **Reduced-optimization local mode (the "works everywhere regardless of GPU" hatch).** `backend.ts`'s `localAllowStockKv()` + `warnReducedOptimizationLocalMode()`, `dflash-server.ts`'s `resolveCacheTypeForBackend()`, and `build-llama-cpp-dflash.mjs`'s `writeCapabilities()` (`reducedOptimizationLocalMode` field + the `ELIZA_DFLASH_ALLOW_REDUCED_KERNELS` opt-in). Default behaviour is unchanged (hard refusal); this is opt-in, loud, and non-publishable. It exists so the §4 voice pipeline *runs* on a backend that hasn't dispatched a required kernel yet — it does not relax the §3 contract for `defaultEligible` bundles.
- **iOS/Android voice path.** The voice bridge takes an in-process text runner via `LocalInferenceEngine.runVoiceTurn({ textRunner })` (the AOSP/Capacitor `AospDflashAdapter.voiceTextRunner()` adapts the libllama-backed `--spec-type dflash` server onto the `DflashTextRunner` contract — streaming SSE deltas as synthesized verifier-accept events). What still needs a real build: `ios-arm64-metal-fused` / `android-arm64-{cpu,vulkan}-fused` `build-llama-cpp-dflash.mjs` targets so the fused `libelizainference` (OmniVoice TTS + Qwen3-ASR) is carried inside the xcframework/AAR (`omnivoice-fuse/prepare.mjs` builds the lib on a macOS/Linux host today but not the iOS/Android cross-targets); the Capacitor `Microphone` plugin + native `AudioTrack`/`AVAudioEngine` sink wiring to `PushMicSource`/`PcmRingBuffer`; `onnxruntime-mobile` (or the Capacitor ONNX bridge) for Silero VAD. Requires an Xcode (`packages/app-core/scripts/ios-xcframework/build-xcframework.mjs` + `run-physical-device-smoke.mjs`) / Android-Studio NDK (`aosp/compile-libllama.mjs`) host. The dflash speculative path through the FFI shim (rather than spawning `llama-server`) — binding `common_speculative_*` through `aosp/llama-shim` — remains "path b" follow-up work; the adapter currently spawns the cross-compiled `llama-server` ("path a") and `voiceTextRunner()` works against either.
- **No `if (!available) return baseline()`.** Per `packages/inference/AGENTS.md` §3, none of the runners above degrade to an unoptimized kernel when hardware is missing — they fail closed. Missing required kernels are a hard error, not a quiet fallback — *except* the explicitly-opt-in, loudly-warned, non-publishable reduced-optimization local mode noted above.
- **`elizaOS/llama.cpp` fork ref.** Repo references point at `elizaOS/llama.cpp`; the GitHub org transfer of the fork repo is complete. `ELIZA_DFLASH_LLAMA_CPP_REMOTE` overrides the clone URL when needed.
- **Verification matrix source of truth.** `packages/inference/README.md` ("Verification matrix") and `packages/inference/verify/kernel-contract.json` (checked by `make -C packages/inference/verify kernel-contract`) are the executable record of which kernel × backend × device combinations actually passed. This ledger is the prose index of the gaps; if it disagrees with the contract gate, the contract gate wins.
