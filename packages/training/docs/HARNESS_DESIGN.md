# GPT-OSS Action-Trajectory Harness — design

## Goal

For every catalog action (currently 111 across 22 plugins, indexed in `data/prompts/actions-catalog.json`), produce a set of **canonical eliza training records** where:

1. The user message is a natural request that should trigger that specific action.
2. The system prompt matches the runtime's `message_handler` template (or `tool_call` template depending on action type).
3. The assistant TOON response is **emitted by gpt-oss-120b under the same prompt the runtime uses**, validated, and saved.
4. The `expectedResponse` exactly matches the runtime decoder's expected schema for that task type.

This closes three gaps the upstream datasets do not cover well:

- **Coverage gap**: 111 catalog actions vs. ~50 distinct action names that show up in the corpus today.
- **Schema-drift gap**: every record in the harness output is by construction in the canonical TOON shape because we generate it under the canonical prompt.
- **Reasoning gap**: the model's own first-person chain-of-thought becomes the `thought:` field, no synth round needed.

## Architecture

```
                                                      ┌────────────────────────────┐
  actions-catalog.json   ──┐                          │ scripts/harness/run.py     │
  prompts/registry.json  ──┼─►  scenario generator ──►│  (async, ~16 concurrent)   │
  persona pool (yaml)    ──┘                          │                            │
                                                      │  for each scenario:        │
                                                      │   1. build canonical msg   │
                                                      │   2. call gpt-oss-120b     │
  scripts/harness/scenario_pool/<action>.jsonl ◄──────│   3. parse TOON response   │
                                                      │   4. validate via decoder  │
                                                      │   5. emit eliza record     │
                                                      └────────────┬───────────────┘
                                                                   │
                                                                   ▼
                       data/synthesized/harness/<action>.jsonl  (canonical eliza records)
                       data/synthesized/harness/manifest.json
                       data/synthesized/harness/failures.jsonl   (rejected outputs)
```

## Components

### 1. Scenario generator (`scripts/harness/scenarios.py`)

For each action in `actions-catalog.json`, produce **20–50 varied scenarios**. A scenario = `(persona, user_message, expected_action_name, expected_arg_keys)`.

Variation axes (combined to fight overfitting):

- **Persona / register**: terse / formal / casual / non-native / multilingual / power-user / first-time-user / sleepy / impatient
- **Phrasing**: imperative ("Create a PR for..."), declarative ("I want to..."), question ("Could you..."), abbreviation ("plz commit and push")
- **Argument completeness**: all required + optional args supplied / required only / required missing (forces clarification reply, not action) / over-specified
- **Language**: en, es, pt, ja, zh, fr, de, ko, ru — proportional to corpus already (so test/val splits stay i.i.d.)
- **Distractors**: same scenario but with text that *looks* like another action ("don't run the migration, just commit") to teach negative routing

Generated by gpt-oss itself: feed the action description + parameters + 3 example_calls and prompt for "give 25 distinct user messages spanning the variation axes that should trigger this action". The generator output goes to `scripts/harness/scenario_pool/<action>.jsonl` — produced once, persisted, reusable.

### 2. Canonical prompt builder (`scripts/harness/prompt.py`)

Reuses `data/prompts/registry.json`. For each scenario:

```python
record = {
    "roomName": "harness-room",
    "agentId": "milady",
    "memoryEntries": persona.memory_entries,  # 0-3 prior turns from persona
    "currentMessage": {"role":"user","speaker":persona.name,"content":scenario.user_message,"channel":"chat"},
    "expectedResponse": "",                   # filled by gpt-oss
    "availableActions": catalog_actions_visible_to_this_persona,
    "metadata": {
        "task_type": "tool_call" if action.is_tool_call else "message_handler",
        "system_prompt": render_handlebars(registry[task_type].template, ctx),
        "toolSpecs": [action_spec],
        "expected_action": action.name,
        "expected_arg_keys": [p.name for p in action.parameters if p.required],
    }
}
```

The system prompt uses the same handlebars template the runtime uses, with `agentName=milady`, `availableActions=...`, etc. — guaranteed runtime parity.

### 3. Generator (`scripts/harness/run.py`)

Async Groq client (re-uses the proven `synthesize_reasoning_groq.py` plumbing — async httpx, retry-with-backoff, TPM throttle).

For each scenario:
1. Render messages: `[{role:"system",content:system_prompt}, ...memoryEntries, {role:"user",content:scenario.user_message}]`
2. Call `openai/gpt-oss-120b` with `reasoning_effort: "low"`, `max_tokens: 800`, `temperature: 0.7`
3. Capture `content` (preferred) or `reasoning` field as the assistant TOON

### 4. Validator (`scripts/harness/validate.py`)

Decode the response via the existing `lib.toon.ToonDecoder` (bun-backed, runtime-parity). Then check:

- decode succeeds
- top-level is a dict
- has required keys for the task_type (per registry)
- `tool_calls[*].name` (or `actions[*].params.tool`) ∈ catalog
- when scenario is "args complete", expected action name is invoked with at least the required keys
- when scenario is "args missing", response is a clarifying REPLY, not the action
- `thought:` is non-empty, ≥5 words, first-person (uses the existing `is_clean_thought` filter)

Failures land in `failures.jsonl` with the raw output + reason. We retry once with `temperature=0.4` before recording failure.

### 5. Emit (`scripts/harness/emit.py`)

For each validated trajectory, emit one canonical eliza record to `data/synthesized/harness/<action>.jsonl` with `metadata.source_dataset = "harness/<action>"` so it dedupes / splits cleanly through the existing pack pipeline.

## Output coverage

Target: **20–50 records per action × 111 actions = 2,200–5,500 records**.

Why this is more valuable than its raw count suggests:

- Each record covers the **exact prompt contract** the runtime uses (`message_handler` / `tool_call` templates) — eliminates schema drift entirely.
- Negative-routing scenarios (e.g., "I'd rather not commit yet") teach the model when *not* to invoke the action.
- Argument-incompleteness scenarios teach the model to fall back to clarifying replies — a behaviour the upstream datasets cover unevenly.
- Persona variation hardens against single-style overfitting that `synth-action-pairs-actions` showed (every record reads "I see the user wants…").

## Cost & throughput

- ~5,000 scenarios × ~600 tokens/req ≈ 3M tokens total.
- Groq gpt-oss-120b: 250k TPM ⇒ ~30 sec per 100 reqs sustained.
- Wall-clock: scenario generation ~30 min, validated emission ~1.5h.
- $cost: trivial against the existing budget (well under $5 at current Groq pricing).

## Integration with the existing pipeline

After harness emits per-action JSONLs:

1. They feed into `scripts/lib/adapters.py` via a new `HarnessAdapter` that just yields the records (they're already canonical).
2. The pack pipeline picks them up, runs through the existing dedupe + group-key assignment + train/val/test split.
3. `data/final/train.jsonl` regenerated with the harness records mixed in. Per-source cap (50k) applies, so harness contributes ~5k unique records — exactly the gap currently filled by short-and-repetitive `synth-*-actions` data.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| gpt-oss emits non-TOON formats (markdown fences, JSON instead of TOON) | Validator rejects, retry at lower temp; second failure goes to `failures.jsonl` for manual review. |
| Model invokes the wrong action on a positive-scenario | Validator catches mismatch, retry once. If still wrong, classify scenario as ambiguous and either drop or relabel. |
| Synthesis bias (gpt-oss's natural style overpopulates the corpus) | Persona pool has at least 30 distinct entries; scenarios sampled stratified across personas at emit time. |
| Coverage gap on actions whose canonical use needs prior context (e.g. `LIST_AGENTS`) | `memoryEntries` populated from a per-action context template (e.g. `LIST_AGENTS` includes a stub history of agent spawns). |
| TPM throttling stalls progress | We already validated at 15-22 rps sustained; harness load is < 5k requests, no concern. |

## Phasing

- **Phase 0 (now)**: scenario generator only. Produces and persists 5k scenario pool. Cheap, parallelizable, idempotent.
- **Phase 1**: run.py + validate.py end-to-end on a single high-leverage action (e.g. `FINALIZE_WORKSPACE`) for QA. Confirm the full pipeline emits a record that passes the existing TOON decoder + schema checks.
- **Phase 2**: full sweep across all 111 actions.
- **Phase 3**: re-pack and (optionally) train v4 on the augmented corpus.
