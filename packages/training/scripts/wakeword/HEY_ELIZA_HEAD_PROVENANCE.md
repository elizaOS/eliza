# "hey eliza" wake-word head — training provenance & evaluation

Reproducible record of training the Eliza-1 default wake-word head ("hey
eliza") on the frozen openWakeWord melspectrogram → embedding front-end, and
its measured accuracy through the real `wakeword-cpp` C runtime (the path the
app actually ships). Companion to `train_eliza1_wakeword_head.py` and
`synthesize_wakeword_corpus.py`.

The trained binaries (ONNX head, GGUFs, corpus WAVs) are NOT committed — they
are multi-MB artifacts reproduced from the scripts below, mirroring the
`voice-models.ts` "download, don't commit" pattern. This file records the
recipe, the corpus sizes, and the held-out numbers so a production publish is
turn-key.

## TL;DR

A real "hey eliza" head **works** and clearly beats the placeholder, but only
once it is trained with the same mel preprocessing the C runtime uses. The key
finding is a train/inference featurization mismatch (below). With that fixed,
the head reaches **~98.7% true-accept / ~3.3% false-accept** on a held-out set
scored through the actual `libwakeword.so`.

The head binary is not yet published into the tier bundles, so `hey-eliza`
stays in `OPENWAKEWORD_PLACEHOLDER_HEADS` until the GGUF below is uploaded to
the bundle catalog. Everything needed to do that is here.

## Front-end (frozen, shipped verbatim)

- `melspectrogram.onnx`, `embedding_model.onnx` — openWakeWord v0.5.1
  (Apache-2.0), `dscripka/openWakeWord`, pinned commit
  `368c03716d1e92591906a84949bc477f3a834455` (matches
  `wakeword_to_gguf.py::OPENWAKEWORD_UPSTREAM_COMMIT`).
- Only the per-phrase classifier head is trained; the two front-end graphs are
  unchanged.

## Corpus (synthesized, piper-tts)

Generated by `synthesize_wakeword_corpus.py` with 12 `rhasspy/piper-voices`
(en_US lessac/amy/ryan/libritts-high/hfc_female/hfc_male/kusal/joe + en_GB
alan/cori/northern_english_male/jenny_dioco — MIT / public-domain-leaning),
swept over length-scale (speed), noise-scale / noise-w (timbre), and
multi-speaker ids; each clip center/random-padded to a 2.0 s window at 16 kHz
mono PCM16 with gain + additive-noise augmentation.

| Split            | positives | negatives | seeds        | trainer sees it? |
| ---------------- | --------- | --------- | ------------ | ---------------- |
| Train            | 4553      | 2670      | 100–107, 200–207 | yes          |
| Held-out (eval)  | 480       | 800       | 900–903      | **no**           |

- Positives: "hey eliza" + natural spelling/punctuation variants.
- Negatives: hard near-phrases over-represented at 40% ("hey alexa", "hey
  lisa", "hey liza", "eliza", "elissa", "elisa", "a pizza", "hey there", …),
  generic English, plus the repo's real-speech fixtures
  (`plugins/plugin-local-inference/native/verify/asr_bench_fixtures/**/*.wav`
  and `omnivoice.cpp/examples/freeman.wav`) chopped into windows (2654 synthetic
  + 16 real-speech-fixture windows in the train split).

## The featurization mismatch (the load-bearing finding)

openWakeWord's Python pipeline rescales the melspectrogram before the embedding
model: `mel = mel/10 + 2`. The `wakeword-cpp` C runtime
(`packages/native/plugins/wakeword-cpp/src/wakeword_runtime.c`) does **not** —
it feeds the raw log-mel straight into the embedding model, and its parity test
(`test/wakeword_parity_test.py`) confirms the C path agrees with a *no-rescale*
ONNX reference. So a head trained with the rescale is featurized differently
than it runs at inference.

Measured on the held-out set through the real `libwakeword.so` FFI
(`verify_ffi.ts`, fresh session per clip, peak P(wake), threshold 0.5):

| Head featurization        | true-accept | false-accept | positive score shape |
| ------------------------- | ----------- | ------------ | -------------------- |
| WITH `mel/10+2` (default) | 30.8%       | 2.1%         | bimodal — fires ~1.0 or ~0 |
| WITHOUT rescale (parity)  | **98.7%**   | **3.3%**     | confident — p10=0.999, p50=1.0 |

(WITH-rescale is the full 480+800 held-out run; WITHOUT-rescale figures are the
150+150 sample — TA holds 97–99% across thresholds 0.3–0.9. Replace with the
full 480+800 number when re-run.)

Root cause cross-checked in pure ONNX (no C runtime): scoring the held-out
positives through the front-end WITH the rescale and the WITH-rescale head gives
100% (mean 0.994); feeding the same head NO-rescale embeddings drops it to
~30% — exactly reproducing the C-runtime result. The fix is `--no-mel-rescale`
in the trainer (`mel_rescale=False`), which featurizes exactly as the deployed
runtime does.

## Reproduce

```bash
# 1. corpus (train + held-out; held-out uses disjoint seeds 900+)
python3 packages/training/scripts/wakeword/synthesize_wakeword_corpus.py \
    --voices-dir <piper-voices> \
    --positives-dir /tmp/wakeword/corpus/pos --negatives-dir /tmp/wakeword/corpus/neg \
    --n-positives 4500 --n-negatives 2600 \
    --extra-negatives plugins/plugin-local-inference/native/omnivoice.cpp/examples/freeman.wav \
      plugins/plugin-local-inference/native/verify/asr_bench_fixtures/non_publish_structure_5utt/*.wav \
    --seed 100
# (repeat to /tmp/wakeword/rate with --seed 900 for the held-out split)

# 2. train WITH C-runtime mel parity
python3 packages/training/scripts/wakeword/train_eliza1_wakeword_head.py \
    --phrase "hey eliza" --no-mel-rescale \
    --positives-dir /tmp/wakeword/corpus/pos --negatives-dir /tmp/wakeword/corpus/neg \
    --front-end-dir <front-end> --out hey-eliza.onnx --epochs 40

# 3. convert to the runtime's three GGUFs
python3 packages/native/plugins/wakeword-cpp/scripts/wakeword_to_gguf.py \
    --melspec-onnx <front-end>/melspectrogram.onnx \
    --embedding-onnx <front-end>/embedding_model.onnx \
    --classifier-onnx hey-eliza.onnx --phrase "hey eliza" --out-dir gguf

# 4. build + verify on the HELD-OUT set through the real FFI
cmake -S packages/native/plugins/wakeword-cpp -B .../build -DCMAKE_BUILD_TYPE=Release
cmake --build .../build -j
bun verify_ffi.ts .../build/libwakeword.so gguf/hey-eliza.{melspec,embedding,classifier}.gguf \
    /tmp/wakeword/rate/pos /tmp/wakeword/rate/neg 0.5
```

## To actually ship it (the remaining step)

1. Publish `hey-eliza.{melspec,embedding,classifier}.gguf` (≈1.5 MB) into every
   tier bundle's `wake/` (or the combined `openwakeword.gguf`) via the training
   publish flow — **the head must be trained `--no-mel-rescale`**.
2. Then remove `hey-eliza` from `OPENWAKEWORD_PLACEHOLDER_HEADS` in
   `plugins/plugin-local-inference/src/services/voice/wake-word.ts`. Until the
   binary is in the bundles, the flag stays (the bundle still carries the
   renamed `hey_jarvis` placeholder).
