| `block_q4_polar` | `PolarQuantResult` has separate `codes: int8` (one per coord), `norms: fp16` (one per block), optional `qjl_signs: uint8` (one bit per block, padded byte) | C: `{ fp16 d; uint8 qs[64]; uint8 qjl[16]; }` packed = 82 bytes per 128-elem block | **Layout mismatch is intentional** — the safetensors sidecar is a flat `{name.codes, name.norms, name.qjl}` map, not a packed-block stream. A GGUF converter would have to (a) re-pack two int8 codes per byte (lo nibble = even index, hi nibble = odd index, matches C `polar_quantize_ref.c`), (b) write fp16 norm at `d`, (c) write the per-block QJL bit at `qjl[0] & 1`. The Python output preserves enough info to do this losslessly. |

### 4. FWHT / Hadamard ordering

| Recipe | Reference | Match? |
| --- | --- | --- |
| `polar_quant._hadamard_matrix(128)` — recursive Sylvester construction, normalized by 1/√2 per recursion step → orthonormal H with `H @ H.T == I` | `polar_hadamard.c::polar_hadamard_inplace(x)` — iterative Cooley-Tukey butterfly **without** any normalization | **AGREE numerically.** Python applies `blocks_norm @ H * sqrt(d)`. C applies the unscaled butterfly to the unit-norm block. Identity: butterfly(x) = x · (√d · H_ortho). Therefore Python's `(unit @ H) * √d` = C's `butterfly(unit)`. The decoders divide by d and √d respectively to compensate, again aligning. ✓ |
| TurboQuant: recipe does not perform FWHT | `turbo_kernels.c::fwht_128` — pre-multiply by SIGNS1, FWHT in-place, post-multiply by SIGNS2, scale by 1/√128 | N/A — recipe defers to runtime library; manifest fragment should record the FWHT scaling convention so a future cross-check can confirm. |

### 5. Norm correction formula

| Recipe | Reference |
| --- | --- |
| PolarQuant Python: stores `||block||₂` as fp16, applies `(centroids/√d) @ H · ||block||` at decode | PolarQuant C: stores fp16 d = `||block||₂`, applies `(1/d) · butterfly(centroids) · ||block||` — **algebraically identical** ✓ |
| QJL Python: stores per-token bf16 `||k||₂` | QJL C: `qjl_block_qjl1_256.norm_bf16 = bf16(||k||₂)` ✓ same convention. Score scale `√(π/2)/proj_dim` is identical. |
| TurboQuant Python: not stored by recipe | TurboQuant C: per-block recon-norm correction `corrected = ||x||₂ / ||recon_centroids||₂`, stored as fp16. Recipe must trust runtime to apply this; sidecar should record that it expects this form. |

| Recipe | Reference | Match? |
| --- | --- | --- |

## Findings ranked by confidence and impact

### HIGH CONFIDENCE — fixed in this audit

**1. Sidecar manifest fragment is incomplete on every recipe.** AGENTS.md §3 mandates each recipe emit a sidecar recording: kernel target, block layout version, codebook hash, expected per-block tolerance. Today's sidecars carry method/paper/library/source_model/hyperparameters but **NO** kernel target, **NO** block layout version, **NO** codebook hash, **NO** tolerance.

Fix: extend the four `*_apply.py` sidecar payloads with a uniform `kernel_manifest` block. Implemented in this PR for all four recipes via `_common.py::kernel_manifest_fragment(...)`.

### RESOLVED 2026-05-10 — PolarQuant PRNG aligned to xorshift32

**2. PolarQuant `_QJL_SEED` PRNG algorithm divergence — RESOLVED.** Per AGENTS.md §3 ("kernel side wins"), Python now uses xorshift32 to match the canonical C kernel. The decision rationale: (a) Mersenne-Twister's 624-int state cannot be replicated on a GPU kernel; (b) xorshift32 is trivially portable across CPU / Metal / Vulkan / CUDA. No Eliza-1 bundle has been published with the legacy MT signs, so encoded artifacts produced by the previous Python path are invalidated — that's expected and acceptable.

Implementation:

- New canonical helper: `scripts/quantization/polar_xorshift32.py` (`polar_xorshift32_signs(n, seed=42)` returns int8 ±1 array; `polar_xorshift32_state_seq(n, seed)` returns the raw uint32 state sequence for white-box tests).
- `polarquant/polar_quant.py` imports `polar_xorshift32_signs` and uses it on both the encode and decode paths (replacing two `torch.randint(seed=42)` call sites). The reconstructed-weight round-trip continues to cancel because encoder and decoder still use the same generator.
- `polar_qjl.c` and `qjl_polar_ref.c::eliza_polar_qjl_signs` are unchanged — both implement the same xorshift32 (`state ^= state << 13; state ^= state >> 17; state ^= state << 5; sign = (state & 1) ? +1 : -1`), seeded with `POLAR_QJL_SEED == 42`, with `state == 0` rewritten to `1`.

Verification:

- `test_polarquant_python_sign_vector_pinned` now compiles `qjl_polar_ref.c`, calls `eliza_polar_qjl_signs`, and asserts the Python helper produces an identical 128-element ±1 sequence.
- `test_polarquant_full_block_parity_against_c_ref` (new) calls `eliza_polar_quantize_row` and the production Python `polar_quantize`, packs the Python output into the C `block_q4_polar` layout, and asserts byte-for-byte (82-byte) equality. Pre-existing decode parity (`test_polarquant_block_dequant_parity_against_c_ref`) continues to pass.

**3. QJL projection-matrix shape convention. — RESOLVED 2026-05-10.**

Original finding: `qjl_apply._build_jl_projections` produced `torch.randn(proj_dim, head_dim, ...)` — i.e. `(proj_dim, head_dim)`. The vendored CUDA/PyTorch consumer (`qjl/qjl_kernel.py::qjl_quantize_pytorch` line 167) expects `(sketch_dim, head_dim)` and transposes internally. The standalone CPU reference at `qjl-cpu/include/qjl/qjl.h` and the verify reference at `inference/verify/qjl_polar_ref.c` document `prj` as `(head_dim, proj_dim)` row-major, i.e. the **transpose**.

**Resolution (2026-05-10):** Python now stores `(head_dim, proj_dim)` row-major, matching the kernel-side canonical layout. The matmul is `q_sketch = q @ pi` (no `.T`), where `q` is `(..., head_dim)` and `pi` is `(head_dim, proj_dim)`, producing `q_sketch` of shape `(..., proj_dim)`. Bytes emitted by `_build_jl_projections` can now be passed directly to qjl-cpu / Metal / Vulkan / CUDA kernels with no transpose at the boundary — that's the whole point of the consolidation.

Layout details (kernel-canonical):
- `pi[i, j]` (row `i`, col `j`) at byte offset `(i * proj_dim + j) * 4`
- one row stride: `proj_dim * 4` bytes (1024 B at proj_dim=256)
- full matrix: `head_dim * proj_dim * 4` bytes (131072 B / 128 KiB at head_dim=128, proj_dim=256)

The legacy CUDA bridge in `qjl/qjl_kernel.py::qjl_quantize_pytorch` still expects `(sketch_dim, head_dim)`. Per the rule that we don't touch the training-side fused CUDA kernel, the recipe transposes once at the bridge boundary (`proj.transpose(0, 1).contiguous()` in `attach_qjl_to_cache`) before handing off to the legacy consumer. The canonical store stays `(head_dim, proj_dim)`.

The stale sidecar field `projection_matrix_layout: "(proj_dim, head_dim)"` has been dropped — there is one canonical layout, no compat shim.

Verification: `test_qjl_projection_layout_matches_c_ref` in `test_recipes_smoke.py` builds Π via `qjl/qjl_quant.build_projection_matrix(head_dim=128, proj_dim=256)`, quantizes a synthetic key with the recipe's helper, pushes the SAME bytes through the C `eliza_qjl_quantize_row` in `inference/verify/qjl_polar_ref.c`, and asserts byte-exact `block_qjl1_256` parity (zero-byte diff). Companion test `test_qjl_apply_recipe_projection_shape_is_canonical` pins the recipe's emitted shape to `(head_dim, proj_dim)` so a regression is caught at unit-test time.

**4. PolarQuant Lloyd-Max iteration count.** The Python recipe iterates 100 rounds by default. At 100 rounds the centroids match the C `POLAR_Q4_CENTROIDS` exactly. At 200+ rounds Python drifts to a different (more converged) solution. The C centroid file is auto-generated at 100 iterations. Both sides are consistent **today**, but if anyone bumps the Python default they will silently break the C pin. Audit recommendation: pin `n_iter=100` and add a regression test that asserts the centroid values match the C reference. Implemented in this PR.

### LOW CONFIDENCE — out of scope

- The vendored `fused_turboquant_vendored.hf.patch_model` is an opaque library entry; this audit did not inspect its kernel source. The recipe's job is to call it and record the config. Manifest fragment work in this PR is sufficient for the audit's scope.

## Tests added

- `test_recipes_smoke.py::test_polarquant_centroids_match_c_reference` — pins Python `_compute_lloyd_max_centroids(16, n_iter=100)` against the canonical 16 floats from `polar_centroids.h`. Tolerance: 1e-6.
- `test_recipes_smoke.py::test_polarquant_qjl_correction_magnitude_matches_c` — pins `_QJL_CORRECTION_MAGNITUDE` to 0.5 and `_QJL_SEED` to 42, matching `POLAR_QJL_CORRECTION_MAGNITUDE` / `POLAR_QJL_SEED`.
- `test_recipes_smoke.py::test_polarquant_python_sign_vector_pinned` — compares the Python `polar_xorshift32_signs(128, 42)` output against the C `eliza_polar_qjl_signs` output byte-for-byte (zero tolerance). Updated 2026-05-10 from the previous MT-pin form when finding (2) was resolved.
- `test_recipes_smoke.py::test_polarquant_full_block_parity_against_c_ref` (new 2026-05-10) — runs the production Python encoder and the C reference encoder (`eliza_polar_quantize_row`) on the same fp32 input block, packs the Python output into the C `block_q4_polar` layout, and asserts byte-for-byte equality on the full 82-byte struct. This is the encode-side companion to the existing decode-parity test.
- `test_recipes_smoke.py::test_polarquant_block_dequant_parity_against_c_ref` — compiles `inference/verify/qjl_polar_ref.c` once via `cc`, dlopens it via ctypes, and checks that the C decoder applied to a synthetic `block_q4_polar` produces the same fp32 vector as the Python decoder applied to the unpacked equivalent (within 1e-3 absolute, which is the published Lloyd-Max bucketization tolerance for 4-bit). Skips with a clear message if `cc` is missing.
- `test_recipes_smoke.py::test_qjl_block_layout_packing_matches_c_ref` — same compile-and-ctypes path. Quantizes a synthetic 128-d key with the recipe's bit-packing convention, then byte-by-byte compares the resulting 34-byte block against `eliza_qjl_quantize_row` from the C reference. Tolerance: byte-exact (zero), since the projection matrix is shared as fp32 between both sides.
- `test_recipes_smoke.py::test_recipe_sidecar_manifest_fragment_complete` — instantiates each recipe, builds a sidecar payload via the new `_common.kernel_manifest_fragment`, and asserts the four AGENTS.md §3 fields are present and non-empty.
- `test_recipes_smoke.py::test_turboquant_recipe_codebook_hash_pinned` — asserts the codebook hash recorded in the manifest fragment matches the SHA256 of the bytes of the canonical `ELIZA_TURBO_*` constants in `turbo_kernels.c`.

The new tests are CPU-only and do not depend on `transformers`, `peft`, or any model checkpoint. The compile-and-ctypes tests skip cleanly when `cc` is unavailable.
