# Eliza-1 optimization inventory

Survey date: 2026-05-11. Dev box: RTX 5080 Laptop GPU (16 GB, Blackwell sm_120,
CUDA 13). Read-only audit — no code changed.

Estimation method. Generation (`tg`) for these models is **memory-bandwidth
bound**: `tg ≈ effective_bw / bytes_streamed_per_token`, where
`bytes_streamed_per_token ≈ model_weight_bytes + KV_bytes_touched`. For a
0.6 B model at short context the weights dominate and KV is noise; for the
27 B-256k/1m tiers the KV term is the same order as the weights. Prefill
(`pp`) is **compute bound** (GEMM throughput → tensor-core util), so quant of
*activations*/attention helps it, weight quant barely does. Where a number is
a measurement it is cited from `THROUGHPUT.md` / the `*_report.json` runs;
everything else is the bandwidth/arith-intensity estimate and is flagged
"(est.)" or "(guess)".

Reference geometry (Qwen3-0.6B, used for the KV math): 28 layers, 8 KV heads,
head_dim 128 → f16 KV = `28·2·8·128·2 = 114,688 B/token` (matches the QJL
report). 27B tiers per `model_registry.py` placeholders: ~64 layers, 4 KV
heads, head_dim 256 → f16 KV ≈ `64·2·4·256·2 = 524,288 B/token` (≈ 64 GB at
128k ctx, ≈ 512 GB at 1M ctx — the reason the long-ctx tiers need KV quant +
spill at all).

## Biggest unrealized wins (ranked)

1. **No CUDA build of the fork → eliza-1 inference falls to Vulkan ≈ 14× slower prefill.** `THROUGHPUT.md` measured Vulkan Q4_K_M at 1,499 t/s prefill vs CUDA ~22,000. The fork's CUDA backend (including the custom `fattn-vec-instance-{qjl,polar,tbq}.cu` kernels) is confirmed to compile with stock nvcc 12.0 at `CMAKE_CUDA_ARCHITECTURES="89-real;90-real;90-virtual"` (driver JITs to sm_120). Action: add a `linux-x64-cuda` fork build target (recipe already in `build-llama-cpp-dflash.mjs`; the target *exists in `SUPPORTED_TARGETS`* — it just isn't being built by the install pipeline, which currently produces `linux-x64-cpu`, `linux-x64-cpu-fused`, `linux-x64-vulkan`). Build with `-j2`, not `-j$(nproc)`, on a ≤32 GB box (fattn TUs OOM-kill at full parallelism). **This is the single highest-leverage fix** — it makes every kernel-level optimization below actually pay off.

2. **Q4_POLAR weight emit is DEFERRED → the 9B/27B tiers ship Q4_K_M and the 0.6B/1.7B tiers ship q8_0.** Q4_POLAR is ~4.25 bpw vs q8_0's 8.5 — closing the converter gap (`Q4_POLAR` in the fork's `gguf-py` + a Python `quantize_q4_polar` packer) would **roughly halve weight bytes vs q8_0 → ~1.8–2× tg for the bigger tiers** (bandwidth-bound regime), with an expected PPL hit of only ~+0.05–0.15 (the WHT rotation makes 4-bit nearly as good as the K-quant). For the 27B tiers: q8_0 ≈ 29 GB of weights → ~20 t/s ceiling on a ~600 GB/s laptop; Q4_POLAR ≈ 15 GB → ~40 t/s. The runtime kernels (`fattn-vec-instance-q4_polar.cu`, `polarquant-cpu`) already exist — only the converter side lags.

3. **Liger kernel can't init on this box (and `python3.12-dev` is missing in general).** The same missing dev-headers package blocks: Liger (Triton `cuda_utils.so`), `fused-turboquant` (Triton), and the QJL CUDA *apply* extension. Without Liger the registry seq-len budgets (4k/8k/16k/65k) don't hold — the fp32 logit transient (`B·S·V·4`, Qwen vocab ~152k) blows VRAM ~4× earlier. Action: `sudo apt install python3.12-dev` (and `nvidia-cuda-toolkit` for `nvcc`). One-line fix, unblocks three optimizations.

4. **Logical batch / `-b 2048` may not be set by the engine by default.** THROUGHPUT.md's best CUDA config is `-ngl 99 -fa 1 -b 2048`; the `optimizations` block in `catalog.ts` has no `batchSize` field, so the dflash-server may be running the node-llama-cpp default. Worth confirming the server is invoked with `-b 2048` for prefill — it's free (no quality cost) and is the difference between ~14k and ~22k t/s prefill on 0.6B (guess on the exact split, but the lever is real).

5. **FP8 training is gated off Blackwell consumer (sm_120) even though sm_120 has native FP8/FP4 tensor cores.** `te_fp8.py` allows only sm_90/sm_100. Extending the arch gate to sm_120 + installing `transformer_engine[pytorch]` would give ~20–30% step-time on the bigger tiers when training on a 5090/Blackwell workstation. Currently the box also lacks TE installed at all, so this is two steps. Lower priority — it speeds *iteration*, not serving.

6. **imatrix calibration should be unconditional for every K-quant publish.** `--calibration` is "accepted for parity" and only forwarded when present — if the publish pipeline isn't always passing a calibration corpus, the shipped Q4_K_M GGUFs are leaving ~0.15 PPL on the table for free (quantize-time only, zero runtime cost).

7. **TurboQuant V-cache and `turbo3_tcq` are runtime-only quants with no fast path on this box.** `turboquant_apply.py` writes a sidecar, not quantized weights; the actual quant happens at inference in `turbokv` (pure-PyTorch, **~3.5–5× slower tg** — measured 66.8→12.2 t/s) or `fused-turboquant` (Triton, BLOCKED on dev headers). For the GGUF runtime the fork's `fattn-vec-instance-tbq*.cu` kernels would be fast — but, again, no CUDA fork build. Note: no current eliza-1 tier actually uses the TBQ V-types (every tier is ctx > 8k → V-cache is `q4_polar`); the TBQ path is dead until the ≤8k branch is used.

8. **The 0_6b drafter companion is dead weight.** `catalog.ts` declares `eliza-1-0_6b-drafter`, but `distill_dflash_drafter.py`/`THROUGHPUT.md` say the 0.6B tier gets no drafter (no smaller Qwen3 base; tg already ~360–440 t/s). Either drop the companion entry or document why it's there. Not a perf loss, just inconsistency that wastes ~0.25 GB of bundle download.
