# EOT Classifier — Publish Gates

The YAML front-matter above is the source of truth for the
`eval_eot_lora.py` gate enforcement. Edit the values there to update
the gates; do NOT duplicate them in code.

## Rationale per gate

### `auroc_min: 0.85`

End-of-turn classification is a binary problem where the **score
ordering** matters more than the absolute threshold — the voice state
machine uses `P(done)` against multiple thresholds (≥0.9 commit early,
≥0.6 enter PAUSE_TENTATIVE, <0.4 extend hangover, see
`eot-classifier.ts` docstring). AUROC measures how well a classifier
ranks positives above negatives; 0.85 is the minimum bar at which
those three thresholds carve out useful regimes.

LiveKit's published turn-detector benchmarks report 0.92-0.95 AUROC
on their internal eval. A LoRA on eliza-1 should clear 0.85 if the
training corpus matches the deployment distribution; if it can't, the
chat-model base is not a good fit for this task and we keep the
LiveKit GGUF path canonical.

### `ece_max: 0.05`

Expected Calibration Error with 10 equal-width buckets in [0, 1].

The voice state machine consumes raw `P(done)` numbers, so a
miscalibrated classifier (e.g. always returning 0.7 for "yes" and
0.3 for "no") would break the hangover logic even at perfect AUROC.

ECE ≤ 0.05 is achievable on a well-trained binary classifier; the
LoRA training objective in `train_eot_lora.py` is direct CE on the
`<|im_end|>` next-token logit, which gives good calibration out of
the box.

### `p95_latency_ms_max: 50.0`

The classifier runs in-line on every partial ASR transcript — the
voice state machine consumes `VoiceTurnSignal` events per partial
(typically every 100-300 ms). A 50ms p95 keeps the classifier on the
under-5%-of-budget side of the per-partial pipeline.

For the LoRA hot-swap path, a single forward pass on a 0.8B-2B-4B
model is ~5-20 ms on consumer GPUs and ~50-150 ms on CPU; the gate
is set with GPU deployment in mind. CPU-only deployments should
override this via env or accept the perf degradation explicitly.

## Cross-classifier comparison

`eval_eot_lora.py` always reports all three classifiers
(Heuristic / LiveKit GGUF / LoRA-on-eliza-1) and gates against the
LoRA result when present, falling back to LiveKit, then Heuristic. The
gate values above are the **canonical** thresholds; per-classifier
reporting gives you visibility into how the LoRA compares to the
established baseline before any cutover.

## Operator override

Pass `--gates /path/to/custom_gates.md` to `eval_eot_lora.py` to
override these defaults for one run. The file must follow the same
YAML-front-matter format. Do NOT edit this file to weaken gates for
a one-off run — the gates here are the publish contract.
