# Eliza-1 inference optimization campaign — consolidated plan

Output of a 12-expert survey (TTFT/prefill · streaming ASR · streaming output · CUDA
kernels · quantization · speculative+schema decoding · memory mgmt · multi-model
orchestration · context length · API/harness · pipeline config space · cross-platform/
installer). Each expert wrote a critical report + a prioritized plan biased toward
**time-to-first-token (TTFT) / time-to-first-audio (TTFA) on slow consumer machines**
(4–8 GB phones, 16–24 GB RTX gaming PCs, CPU fallback). This file is the cross-reviewed
synthesis: the shared modules the plans converge on, the unified roadmap, the
conflicts resolved. Per-domain detail also lives in `THROUGHPUT.md`,
`OPTIMIZATION_INVENTORY.md`, `CUDA_KERNEL_PUNCHLIST.md`, `APOLLO_TUNING.md`,
`CONTEXT_SCALING.md` (`packages/shared/src/local-inference/`),
`docs/training/{cuda-setup-and-degradation,schema-constrained-decoding}.md`.

## Executive findings

1. **TTFT is bimodal by backend, not gradual.** On the CUDA fork build a 1k-token system prompt prefills in ~40 ms — TTFT is then dominated by fixed overhead (dispatcher routing, per-call file I/O, the chat-template render, the HTTP round-trip). On Vulkan it's ~14× worse (~600 ms). On CPU it's ~30 s at ~57 t/s — *catastrophic*, and the only lever is **not re-prefilling** (KV-prefix caching is existential there, not a nicety). So "TTFT work" splits: on GPU shave fixed overhead + keep the model & system-KV resident; on CPU/phone the KV-prefix cache is the whole game.
2. **The single biggest cross-platform win: ship a CUDA (and Vulkan/Metal/ROCm) fork build as a release artifact.** eliza-1's GGUFs use the fork's custom GGML types (`Q4_POLAR`/`QJL1_256`/`TBQ`), so they route to the fork's `llama-server` — which the build pipeline produces in CI (`local-inference-matrix.yml`) but **never promotes to a release**, so a fresh NVIDIA install falls to the Vulkan fork build (≈14× slower prefill) or errors. Fix = a `dflash-binaries` release job + a runtime `fetch-binary` resolver in `dflash-server.ts` (`accelBackendKey` already prefers a present-on-disk `…-cuda` build). Turns the dominant desktop case from "broken or 14× slow" to "fast out of the box".
3. **CUDA graphs are already ON** in the fork build (`GGML_CUDA_GRAPHS=ON`) — the usual #1 TTFT item is done. The remaining kernel wins are steady-state `tg`: decode-once-into-shared for the TBQ KV in flash-attn (the TBQ decode currently re-runs a size-32 WHT ~8–16× per K vector), a *fused* QJL flash-attention kernel (`qjl_score_kernel` today materializes the full score matrix in HBM — defeats flash-attn at exactly the long-context regime QJL targets), and `Q4_POLAR` weights once the converter emits them.
4. **`Q4_POLAR` weight-emit is the biggest unrealized quant win** — the runtime kernels exist, but the fork's `gguf-py`/`convert_hf_to_gguf.py` don't emit `q4_polar`, so `gguf_eliza1_apply.py` falls back to q8_0 (`weight_quant.deferred:true`). 4.25 bpw vs 8.5 → ~½ weight bandwidth → ~2× `tg` for 4b/9b/27b; ~½ GGUF download for 0.6b/1.7b.
5. **The streaming voice pipeline is architecturally complete on the fused in-process path** (drafter ∥ verifier overlap, drafted tokens speculatively pushed to a 4-deep phrase chunker, rollback coupled to DFlash rejects, barge-in abort threaded to LLM+TTS). Two holes keep TTFA high: (a) OmniVoice still synthesizes a *whole phrase* before any PCM comes out — wire `synthesizeStream`; (b) the deterministic envelope scaffold (`{"thought":"`, the action-name skeleton, …) is still spent as forward passes because the fork doesn't honor `eliza_prefill_plan` yet. Also: the **desktop *chat* path doesn't use the phrase scheduler at all** (it does a separate cloud-only first-sentence TTS).
6. **ASR runs on a *completed* utterance, not a live stream.** A `StreamingTranscriber` *contract* exists but the production caller batch-feeds the whole VAD-gated clip. The win: chunked-encoder streaming ASR (~300–500 ms windows) → stable partials → the turn controller *prefills the LLM on the stable transcript prefix while the user is still talking* → on `speech-end` the first token lands ~tens of ms later instead of after a multi-second serial ASR+prefill chain. The "speculative LLM on partial" plumbing (`VoiceTurnController.startSpeculative`) already exists; what's missing is real fast partials + making the prefill-on-prefix a cheap KV-warm.
7. **The multi-stage LLM pipeline does N separate prefills** (should_respond → context_routing → action_planner → response → evaluators), each re-prefilling the shared system+history. Wins: combine should_respond+planner into one call; shared-prefill the stages (one KV, branch at the suffix); move evaluators fully off the critical path (they also contend for the GPU with the next turn's first token on a single-GPU box).
8. **The harness does file I/O on every model call** (`readRoutingPreferences`, `readEffectiveAssignments` ×2, `listInstalledModels` disk scan) + a double chat-template render + an eager `stringifyStructuredForPrompt` for logging + a sync `structuredClone` for the trajectory — none memoized. On a slow phone that's tens of ms of pure overhead × 3 calls × every turn.
9. **No memory-aware dynamic context selector.** `dflash.contextSize` is the static catalog `contextLength`; a 0.6b on a 24 GB GPU runs at 32 k leaving ~22 GB idle, while a 4 GB phone running the 1.7b + ASR + TTS + VAD may not fit at 32 k. The fix (computed `maxFittingContext = (vram − weights − secondaries − workingSet) / kvBytesPerToken`, clamp to `min(tier.contextLength, baseNativeMaxPos=40960 for Qwen3)`) is documented in `CONTEXT_SCALING.md` but not built. **Do not bump the catalog `contextLength`** for 0.6b/1.7b — Qwen3's `max_position_embeddings` is 40960; past that needs a RoPE/YaRN-extended GGUF.
10. **No central "device + config → resolved runtime config" module.** The memory, context, and pipeline-config experts independently converged on the same need: one place that takes (probed VRAM/RAM, tier, active config = text/embed/voice-in/voice-out/vision) and emits `{contextSize, cacheTypeK/V, gpuLayers, parallel, drafterChoice, spillArmed, richness}`. Today these decisions are scattered (prewarm in dflash-server, parallel TTS in voice, KV layout in catalog, eviction in memory-monitor) and only grow one direction (never shrink for a tight box).

### Conflicts resolved in cross-review

- **Two homes for the central decision module** (memory expert: fold into `ram-budget.ts`; context/pipeline-config experts: a new `config-policy.ts`/`runtime-config.ts`). → **Resolution:** a thin new `runtime-config.ts` for the *descriptor* (what config is active) + a new `config-policy.ts` for the *resolution* (`planRuntimeConfig`); `ram-budget.ts` keeps the *budget arithmetic* and `config-policy.ts` calls into it. Three distinct concerns, one seam each — no parallel implementations.
- **`/v1/chat/completions` vs `/completion`** (harness expert wants `/completion` raw-prompt to avoid the double Jinja wrap; structured-decode expert's `eliza_prefill_plan`/grammar fields are wired on `buildChatCompletionBody`). → **Resolution:** the fork's `/completion` must accept the same prefill/grammar/skeleton fields as `/v1/chat/completions` before flipping the default; until then keep `/v1/chat/completions` but stop the runtime-side `stringifyStructuredForPrompt` double-render. Tracked as a fork item under P1-6.
- **Bump the catalog `contextLength`?** (a naïve reading of `CONTEXT_SCALING.md`'s "0.6b fits 674k of KV on 16 GB" tempts a huge bump). → **Resolution:** NO — Qwen3's `max_position_embeddings` is 40960; past that the model outputs garbage without a RoPE-extended GGUF. The win is the *runtime* `config-policy.ts` selector stretching `contextSize` toward 40960 on roomy devices and clamping it down on phones, not the static default. (P0-5, P2-9.)
- **0.6b DFlash drafter?** → **Resolution:** NO drafter for the 0.6b tier — no smaller-than-itself Qwen3 base, a non-Qwen drafter has a tokenizer mismatch that breaks verification, and at ~400 tg it's memory-bandwidth bound anyway. The catalog's `eliza-1-0_6b-drafter` companion is dead weight (remove it). Drafters are for 1.7b+ (Qwen3-0.6B drafter) and 9b/27b (Qwen3-1.7B drafter).
- **FP8 training / inference** → **Resolution:** out of scope for the consumer-machine target — `te_fp8.py`'s sm_120 gate is intentional (consumer Blackwell FP8 in Transformer Engine silently falls back to bf16), and FP8 inference is a vLLM/TRT-LLM cloud-serving concern, not the llama.cpp on-device path. Leave as-is.

## What this campaign (training-lane + measurement + docs) executes vs. hands off

**We execute (in-lane, no GPU/hardware/swarm-collision dependency):** P1-11 (imatrix + curated calibration set), the training-side of P0-2 (flip `gguf_eliza1_apply.py` once the converter lands — already wired to detect it), the training-side of P1-7 (`distill_dflash_drafter.py` runs once a GPU is free — the 0.6b/1.7b SFTs are done; the bigger drafters need cloud GPUs), the training-side of P2-4/P2-5 (the corpus-validator arg gate, the compact-schema synth regeneration), `emit_native_grammar.py` (done), the `eliza-native-contract.ts` JSON-schema artifact + the `native_record.py` validation against it. Plus: keeping these docs current as the swarm executes.

**Hands off to the inference team / voice swarm** (their active `packages/inference/` + `local-inference/` + `voice/` rewrite — these are the punch-list, coordinate, don't collide): P0-3, P0-4 (fork side), P0-6, P1-1, P1-2 (fork side), P1-3, P1-4, P1-5, P1-9, P1-10, P2-1, P2-2, P2-3, P2-6, P2-7, P2-8, P2-12, P2-14, and the harness side of P0-5/P1-6 (the `runtime-config.ts`/`config-policy.ts`/`gpu-arbiter.ts` modules + the per-call memoization).

**Hands off to CI/release + UI + mobile:** P0-1 (the `dflash-binaries` release job), P1-8 (the GPU-detect installer flow), P2-10 (mobile SSE coalescing), P2-11 (ROCm/Android/iOS build targets).

**Needs hardware we don't have here:** the 9b/27b SFTs + their drafter distillations (cloud GPUs — `train_vast.sh`/`train_nebius.sh`, commands in `MODELS_STATUS.md`); the `eliza1_gates.yaml` numeric thresholds (real devices — mobile RSS, thermal, ASR WER, voice latencies); `cuda-toolkit-12-8` for the native sm_120 build (P2-12 — one `sudo apt-get install`); `python3.12-dev` for Liger (the 1.7b SFT at seq 4096 needs it; one `sudo apt-get install`).
