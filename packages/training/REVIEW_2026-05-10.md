## HIGH-CONFIDENCE — fixed in this pass

- **Python publishedAt regex over-permissive vs Zod (schema drift).**
  The TS Zod validator's `.datetime()` rejects timezone offsets; the
  Python regex accepted `+00:00`. Tightened Python to `Z`-only with
  parity tests on both sides. Files:
  - `scripts/manifest/eliza1_manifest.py`
  - `scripts/manifest/test_eliza1_manifest.py`

## MEDIUM-CONFIDENCE — surfaced, not auto-fixed

- **`orchestrator.py:632` aliases `e2e_loop_ok` to the `thirty_turn_ok`
  gate.** The manifest schema declares `evals.e2eLoopOk` and
  `evals.thirtyTurnOk` as independent booleans (per AGENTS.md §6).
  Currently both are sourced from the same gate; `e2eLoopOk` is never
  independently observed. Either define an `e2e_loop_ok` gate +
  result key (and add it to the eval-blob shape in
  `eliza1_gates.py`'s docstring), or collapse the two manifest fields
  into one. Right now this is a defensive fallback that obscures the
  fact that one of the two contract gates has no measurement.

- **`orchestrator.py:_required_kernels_for` and
  `_collect_files_for_manifest` both parse `<num>k` from filename
  stems**, with two slightly different parsers. Consolidate into a
  single `parse_text_ctx_from_filename(p: Path) -> int | None` helper
  in the manifest module so the publish gate and the manifest builder
  agree byte-for-byte on long-context detection.

- **`orchestrator.py:339` shape:** `subprocess.run` results from
  `_run_reference_test` are concatenated into the error message
  unbounded. A 100MB stdout from a misbehaving make target lands in
  the OrchestratorError message and then in the log. Trim or truncate.

- **`_common.py:107-124` weak typing on `model_config: object`** — the
  helpers reach for `.head_dim`, `.num_attention_heads`, `.layer_types`
  on an `object`. This is a transformers `PretrainedConfig`; either
  import the type guard or define a `Protocol` so a wrong-arity call
  fails at type-check time. Currently the only safety net is the
  runtime AttributeError.

- **`polar_xorshift32.py:62-66` tight Python loop over uint32** for the
  per-row sign vector. Fine for unit tests; for real publish runs over
  millions of blocks this is the bottleneck. Vectorize with the standard
  trick of pre-computing seed-sequence chunks once per layer. Out of
  scope for the audit but flag for the next quantization perf pass.

- **`fused_turboquant_apply.py` and `turboquant_apply.py` share ~60%
  surface** — same arg parser shape, same calibration-prompts loader,
  same sidecar layout with a different `kernel_manifest` method name.
  AUDIT_2026-05-10.md acknowledges the fused recipe wraps the same
  kernel; consolidate the CLI scaffolding into `_common.py` and have
  each recipe declare only its method-specific bits (sidecar payload
  body, model-loader kwargs).

## LOW-CONFIDENCE / needs human decision

- **`orchestrator.py:899` versioning fallback `return "1.0.0"`.** Reading a
  missing `bundle/VERSION` and silently defaulting to `1.0.0` will
  produce a publish whose tag claims `eliza-1-<tier>-v1.0.0` regardless
  of the real upstream lineage. Consider failing-loud when VERSION is
  missing — operators who mean to publish 1.0.0 can write it.

- **`benchmarks/eliza1_gates.yaml`** marks 32 of 36 numeric gates
  `provisional: true` (every metric except barge-in, mobile-rss, and
  thermal). The publish gate still blocks on them. That's correct per
  AGENTS.md §6 ("provisional gates still gate"), but it means the
  first real publish will almost certainly trip a provisional that was
  set without measurement. Worth surfacing in the publish-runbook.

- **JSON Schema sibling
  `eliza-1.manifest.v1.json`** declares `publishedAt` as
  `format: "date-time"`, which per JSON Schema spec accepts offsets.
  Now in disagreement with both Zod (`Z`-only) and Python (`Z`-only).
  External consumers of the JSON Schema will accept manifests the
  runtime rejects. Consider tightening the JSON Schema with an explicit
  `pattern` mirroring the Python regex, or document the drift.
