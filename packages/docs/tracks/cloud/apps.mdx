title: Cloud Apps
description: "Register a Cloud app, route app-scoped chat, manage domains, and monetize usage."
---

An **app** in Cloud is a registered product record that bundles metadata,
allowed browser origins, hosting/domain settings, analytics, and monetization.
Registering an app gives you an `id` that:

- Identifies the app to Cloud-routed inference, billing, and analytics.
- Lets your backend call app administration APIs.
- Enables app-scoped chat, container deploy metadata, and domain binding.
- Powers creator monetization (earnings tied to app usage).

## Register

`POST /api/v1/apps`

Body:

```json
{
  "name": "My App",
  "app_url": "https://placeholder.invalid",
  "allowed_origins": ["https://placeholder.invalid"],
  "description": "What it does.",
  "skipGitHubRepo": true
}
```

Response includes the assigned app `id` and a one-time app API key. Store the
key server-side. For user-facing app chat, forward the user's bearer token to
`POST /api/v1/apps/{id}/chat` so the user's organization is charged.

## Lifecycle

- `GET /api/v1/apps/{id}` - fetch.
- `PATCH /api/v1/apps/{id}` - update metadata, URLs, origins, and settings.
- `PUT /api/v1/apps/{id}/monetization` - configure creator markup.
- `POST /api/v1/apps/{id}/chat` - run app-scoped chat.
- `DELETE /api/v1/apps/{id}` - delete and clean up.

## Monetization

Apps can earn creator revenue from app-scoped inference markup. The current
monetization payload uses flat fields:

```json
{
  "monetizationEnabled": true,
  "inferenceMarkupPercentage": 100,
  "purchaseSharePercentage": 10
}
```

Older `enabled` plus nested `pricing` payloads are stale and should not be
used.
