## HIGH-CONFIDENCE — fixed in this pass

- **Manifest schema drift between Python and TS validators (publishedAt).**
  Python `_DATETIME_RE` accepted timezone offsets (`+00:00`); Zod's
  `.datetime()` default rejects them. Tightened the Python regex to `Z`-only
  and added parity tests on both sides. Files:
  - `eliza/packages/training/scripts/manifest/eliza1_manifest.py`
  - `eliza/packages/training/scripts/manifest/test_eliza1_manifest.py`
  - `eliza/packages/app-core/src/services/local-inference/manifest/manifest.test.ts`

## MEDIUM-CONFIDENCE — surfaced, not auto-fixed

- **`recommendation.ts:99` weak cast `hardware.platform as string`** to compare
  against `"android"` / `"ios"` which are not in `NodeJS.Platform`. Mobile
  detection effectively relies on the test suite lying about the type
  (`platform: "android" as NodeJS.Platform`). The right fix is to read
  `hardware.mobile?.platform` (which IS typed `"ios" | "android" | "web"`)
  and remove the cast. Pre-existing pattern; touches recommendation.test.ts.

- **`catalog.ts` hardcodes the five tier id strings** (`eliza-1-lite-0_6b`, …)
  and a parallel `recommendation.ts` ladder hardcodes them again. The
  manifest module already exports `ELIZA_1_TIERS`. The catalog should
  derive `ELIZA_1_TIER_IDS = ELIZA_1_TIERS.map((t) => `eliza-1-${t}`)`
  and the ladders should reference those constants. Mechanical change but
  large surface; needs a follow-up sweep across the engine.

- **`runtime-mode.ts:53` typing of `RuntimeModeConfigShape.cloud`** is
  `{ enabled?: unknown } | unknown` — the union is meaningless and forces
  the `typeof config === "object" && "cloud" in config` dance below. Tighten
  to `{ enabled?: boolean } | undefined` once you've audited writers of the
  config (likely safe — `cloud.enabled` is the only consumed field).

- **`runtime-mode.ts:103`** casts `loadElizaConfig() as RuntimeModeConfigShape`.
  `loadElizaConfig` returns `unknown` from `@elizaos/agent`. Either narrow at
  the boundary with a Zod parse or export a typed loader from
  `@elizaos/agent` and drop the cast. This is the boundary the AGENTS.md
  "validate at the boundary, type the validated result" rule names.

- **`remote-forwarder.ts:82-90` request-header copy** silently drops
  array-valued headers (`if (typeof value === "string")`). Cookies and
  `set-cookie` arrive as arrays in node http; if any forwarded route needs
  cookies, they vanish. Document or join with `; ` per RFC 7230.

## LOW-CONFIDENCE / needs human decision

- **`omnivoice-fuse/cmake-graft.mjs` uses `file(GLOB)` with
  `CONFIGURE_DEPENDS`.** This is the standard CMake anti-pattern (won't
  detect new files between configures unless you re-run cmake). If the
  fused build is re-run from scratch on each release, fine; if it's
  incremental, prefer an explicit source list. Out of scope for this
  cleanup pass — touches build infra.

- **`route-mode-matrix.ts` is documented as "first match wins" with prefix
  semantics**, and the comment in `ROUTE_MODE_MATRIX` requires more-specific
  paths above prefixes. There is no test that asserts this ordering
  invariant — a future contributor could add `/api/cloud/` before
  `/api/cloud/billing/` and silently change semantics. Worth a structural
  test.
