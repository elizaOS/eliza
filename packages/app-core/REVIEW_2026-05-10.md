## HIGH-CONFIDENCE — fixed in this pass

- **Manifest schema drift between Python and TS validators (publishedAt).**
  Python `_DATETIME_RE` accepted timezone offsets (`+00:00`); Zod's
  `.datetime()` default rejects them. Tightened the Python regex to `Z`-only
  and added parity tests on both sides. Files:
  - `eliza/packages/training/scripts/manifest/eliza1_manifest.py`
  - `eliza/packages/training/scripts/manifest/test_eliza1_manifest.py`
  - `eliza/packages/app-core/src/services/local-inference/manifest/manifest.test.ts`

## MEDIUM-CONFIDENCE — fixed in follow-up pass (2026-05-10)

- **`recommendation.ts:99` weak cast `hardware.platform as string`.**
  DONE 2026-05-10: read `hardware.mobile?.platform` (typed `"ios"|"android"|"web"`)
  for mobile detection; dropped the cast on `hardware.platform`. Updated mobile
  test cases to populate `mobile: { platform: ... }` and added an iOS-mobile
  classification test.

- **`catalog.ts` hardcodes the five tier id strings.**
  DONE 2026-05-10: `ELIZA_1_TIER_IDS` is now `ELIZA_1_TIERS.map(t => \`eliza-1-${t}\`)`,
  derived from the manifest module's source-of-truth list. The
  `recommendation.ts` ladder definitions reference local
  `Eliza1TierId`-typed constants (`TIER_LITE`/`TIER_MOBILE`/…) so adding
  a tier picks up automatically.

- **`runtime-mode.ts:53` `RuntimeModeConfigShape.cloud` was a meaningless
  union.** DONE 2026-05-10: replaced with a proper Zod schema
  (`RuntimeModeConfigSchema` with `cloud: z.object({ enabled: z.boolean().optional() }).optional()`).
  The `typeof config === "object" && "cloud" in config` guard is gone;
  the resolver reads `config?.cloud?.enabled === false` directly.

- **`runtime-mode.ts:103` `loadElizaConfig() as RuntimeModeConfigShape`.**
  DONE 2026-05-10: replaced both casts with `parseRuntimeModeConfig(loadElizaConfig())`,
  which `safeParse`s through the Zod schema and returns `null` on
  unknown shapes. Validates at the boundary per AGENTS.md.

- **`remote-forwarder.ts:82-90` request-header copy silently drops arrays.**
  DONE 2026-05-10: extracted `buildForwardHeaders(...)` (now exported for
  test) that uses `Headers.append(name, v)` per element of array-valued
  headers (RFC 7230 §3.2.2). Added vitest cases asserting `set-cookie`
  array values survive via `getSetCookie()`, hop-by-hop headers are
  stripped, and `host`/`authorization` are rewritten correctly.

## LOW-CONFIDENCE / needs human decision

- **`omnivoice-fuse/cmake-graft.mjs` uses `file(GLOB)` with
  `CONFIGURE_DEPENDS`.** This is the standard CMake anti-pattern (won't
  detect new files between configures unless you re-run cmake). If the
  fused build is re-run from scratch on each release, fine; if it's
  incremental, prefer an explicit source list. Out of scope for this
  cleanup pass — touches build infra.

- **`route-mode-matrix.ts` is documented as "first match wins" with prefix
  semantics**, and the comment in `ROUTE_MODE_MATRIX` requires more-specific
  paths above prefixes. There is no test that asserts this ordering
  invariant — a future contributor could add `/api/cloud/` before
  `/api/cloud/billing/` and silently change semantics. Worth a structural
  test.
