Original Failure Review

Eliza: why it failed originally

The bridge reset its fake backend every turn because it keyed sessions by id(list(history)), but the runner passes a new list each turn. Tool results disappeared, so Eliza kept waiting and repeating tools.

The prompt also lacked the benchmark clock/world snapshot and duplicated context heavily. Relative dates drifted to 2026-05-12, while ground truth expected 2026-05-11 or 2026-05-10.

Hermes: why it was middle

Hermes had better terminal behavior on the simple create case. It used tools once, then replied. On reschedule, the executor gave no useful search results and rejected recoverable shapes, so Hermes searched repeatedly or used an event title as an id.

OpenClaw: why it looked better

OpenClaw's text protocol encouraged visible reasoning and user-facing confirmations, so it earned output substring credit even when state was wrong. In the old reschedule trajectory, it created a duplicate and tried deleting a hallucinated id, but the prose contained the expected words.

Fixes Implemented

Eliza bridge and runtime

LifeOps sessions are stable for the whole scenario, LifeOps context is rendered once through the provider, and direct REPLY/RESPOND actions are treated as terminal messages instead of fake backend tools.

The benchmark plugin now registers LifeOps compatibility action names so raw planner emissions like CALENDAR are captured instead of surfacing as "Action not found".

Executor and fake backend

Calendar aliases now support action/operation discriminators, fuzzy title lookup, non-id eventId values, camelCase newStart/newEnd, nested details, and nested updates.

Calendar search now returns matching events, not a no-op payload. This is what allowed Hermes and OpenClaw to find event_00040 instead of guessing.

Output scorer

The reschedule scenario now requires 3pm rather than the brittle bare token 3. Output matching normalizes explicit clock times, so 3pm, 3 p.m., 15:00, and 15:00 UTC compare as the same required fact.

The same pass also normalizes Unicode hyphen variants in scenario-authored output strings, which avoids false misses such as wind‑down versus wind-down.

Critical Assessment

OpenClaw was "better" in the original numbers because it was benchmark-friendly, not because it was actually more correct. It narrated its intent, included plausible confirmations, and used a tool-call text protocol that preserved conversational output. That got it output-substring points while its world mutation was still wrong.

Hermes was middle because it had the right high-level discipline on simple tasks: tool first, then a concise confirmation. Its weakness was recovery from missing tool feedback. Without concrete search results and tolerant executor schemas, it repeated searches and never got to a valid reschedule.

Eliza had the strongest production architecture but the weakest harness integration. Once the harness stopped resetting state, supplied the benchmark clock/world snapshot, accepted the planner's actual field shapes, and stopped treating replies as tools, Eliza became the only system to hit a clean 1.000 with the production-shaped runtime path.
