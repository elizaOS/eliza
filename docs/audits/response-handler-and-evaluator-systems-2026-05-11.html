Response Handler and Evaluator Systems Review

A code-level review of the Stage-1 response handler, the response-handler field registry, the post-parse response-handler evaluators, the in-loop trajectory evaluator, and the registered post-turn evaluator service in this repository as of May 11, 2026. Updated after implementation: the field registry is now the production Stage-1 schema and dispatch path.

Executive Summary

What Exists

The repository has four related but distinct systems:

Stage-1 response handler responseHandlerEvaluators responseHandlerFieldEvaluators post-turn EvaluatorService planner-loop evaluator

The production message path now composes the HANDLE_RESPONSE schema from ResponseHandlerFieldRegistry, renders registered prompt slices, dispatches parsed fields to handlers, and then routes to direct reply or planner loop.

Main Finding

The earlier gap has been closed: message.ts calls composePromptSlices(), sends composeSchema() to createHandleResponseTool(), and calls dispatch() before routing.

Active scope LifeOps threadOps is now a field evaluator only. The deleted post-parse work-thread regex router is no longer exported, tested, or benchmarked.

Practical implication: cloud and local providers now see the same field-registry envelope. The legacy HANDLE_RESPONSE_SCHEMA remains only as fallback compatibility when no registry fields are available; normal runtime registration uses the composed all-required schema.

Stage-1 Response Handler

The production response handler is the first model call for a normal inbound message. It asks the model to call HANDLE_RESPONSE once and produce a flat, all-required field-registry envelope.

Actual Tool Shape

The production schema is generated from ResponseHandlerFieldRegistry.composeSchema() and passed into createHandleResponseTool(). The active envelope is:

{
  shouldRespond: "RESPOND" | "IGNORE" | "STOP",
  contexts: string[],
  intents: string[],
  replyText: string,
  threadOps: ThreadOp[],
  candidateActionNames: string[],
  facts: string[],
  relationships: [{ subject, predicate, object }],
  addressedTo: string[]
}

The legacy nested/direct schemas are still present as defensive fallback shapes, but the normal runtime path sends the composed registry schema.

Flow Control

Role and context catalog. runV5MessageRuntimeStage1() resolves sender role and available context definitions.
Prompt render. renderMessageHandlerModelInput() builds the Stage-1 prompt and appends the active field evaluator prompt slices.
Tool definition. createHandleResponseTool() receives responseHandlerFieldRegistry.composeSchema().
Local grammar. buildResponseGrammar() builds local structure forcing from the same registered field envelope.
Model call. Runtime calls ModelType.RESPONSE_HANDLER with tool choice required, max tokens 1024, and optional structured streaming.
Parse and dispatch. The tool arguments are normalized, then responseHandlerFieldRegistry.dispatch() parses and handles active fields in priority order.
Fallback. If no structured field object can be parsed, the runtime falls back to the legacy parser and simple direct-reply fallback.
Post-parse patchers. runResponseHandlerEvaluators() runs deterministic post-parse evaluators unless a field handler preempted the turn.
Route. routeMessageHandlerOutput() decides terminal ignore/stop, final direct reply, or planner needed.
Side memory. Facts and relationships from extract launch a background facts/relationships stage; addressedTo is applied as relationship edges.

Production Routing Outcomes

Parsed planRuntime result
IGNORETerminal silent ignore.
STOPTerminal stop signal.
contexts = ["simple"] or no planning contextEmit replyText directly, with a fallback direct-reply model call only if no usable reply text exists.

Post-Parse Response-Handler Evaluators

This is the deterministic repair system that runs after field-registry dispatch, unless a field handler preempts the turn. It is defined by ResponseHandlerEvaluator in response-handler-evaluators.ts. These evaluators do not influence the Stage-1 model schema or prompt. They inspect and mutate the parsed MessageHandlerResult deterministically.

Registered Evaluators

NameOwnerPriorityWhat it does
core.known_tool_request_repair core, passed as built-in from message.ts 20 Regex/heuristic repair for explicit known tool requests. Forces requiresTool, clears direct reply, and adds contexts/action hints for approvals, credentials, check-ins, calendar/calendar travel/calendly, owner preference, message reply drafting, screenshot requests, and similar known cases.
owner.profile_extraction app-lifeops 30 Owner-only deterministic extraction. Parses names, location, timezone, partner, platform handles, and relationships via regex, writes owner facts/entities/relationship edges, and adds a context slice.

Patch Model

Each evaluator returns a ResponseHandlerPatch. The runner applies patches in priority order and records trace entries for changed fields.

type ResponseHandlerPatch = {
  processMessage?: "RESPOND" | "IGNORE" | "STOP";
  requiresTool?: boolean;
  simple?: boolean;
  setContexts?: AgentContext[];
  addContexts?: AgentContext[];
  addCandidateActions?: string[];
  addParentActionHints?: string[];
  addContextSlices?: string[];
  clearReply?: boolean;
  reply?: string;
  debug?: string[];
};

Context additions are filtered against availableContexts. Candidate actions and parent hints are merged case-insensitively. Errors are logged and do not abort the full message pipeline.

Planner-Loop Evaluator

This is not a plugin registry system. It is a built-in trajectory decision call inside runPlannerLoop(). After planner tool execution, the loop asks an evaluator model whether the task is done, whether one queued tool should run next, or whether the planner must replan.

Control Flow

Planner emits tool calls. callPlanner() builds planned tool calls or a terminal message.
Tools execute. The loop executes queued tool calls and records each step in the trajectory.
Optional gate. tryGateEvaluator() can synthesize a successful FINISH and skip the evaluator model call when a successful tool drained the queue and the planner already supplied a clean explicit messageToUser.
Evaluator call. runEvaluator() uses ModelType.RESPONSE_HANDLER by default with evaluatorSchema.
Decision. FINISH returns final message, NEXT_RECOMMENDED selects a queued tool, CONTINUE clears queue and replans.
Repair. If evaluator says FINISH after tool use without messageToUser, the loop appends an instruction and evaluates once more.

Schema

{
  success: boolean,
  decision: "FINISH" | "NEXT_RECOMMENDED" | "CONTINUE",
  thought: string,
  messageToUser?: string,
  copyToClipboard?: { title: string, content: string, tags?: string[] },
  recommendedToolCallId?: string
}

The evaluator has built-in sanitization for user-visible messages: it strips internal PTY session IDs, generated task-agent labels, and similar orchestration artifacts before delivering messageToUser.

Post-Turn Evaluator Service

This is the registered plugin evaluator system. It runs after the agent response is complete, before ALWAYS_AFTER actions, via runPostTurnEvaluators() in DefaultMessageService. It is separate from the in-loop planner evaluator.

Evaluator Contract

interface Evaluator<TOutput, TPrepared> {
  name: string;
  description: string;
  priority?: number;
  providers?: string[];
  schema: JSONSchema;
  shouldRun(context): Promise<boolean>;
  prepare?(context): Promise<TPrepared>;
  prompt(context): string;
  parse?(output): TOutput | null;
  processors?: EvaluatorProcessor[];
}

Service Flow

Collect. Sort runtime.evaluators by priority then name.
Gate. Run all shouldRun() checks in parallel. Failures become non-fatal errors.
Provider state. Compose state for union of all active evaluator providers.
Prepare. Run each active prepare() in parallel. Failures remove that evaluator from the model call.
One model call. Build one merged JSON schema with one top-level property per evaluator and call ModelType.TEXT_LARGE with JSON output.
Parse and process. For each evaluator section, parse output, then run processors in priority order. Processor failures are isolated.

Key Architectural Difference

Response-handler evaluators are about current-turn routing. They operate before the route to direct reply or planner is finalized.

Post-turn evaluators are about memory, extraction, reflection, and learning. They run after the response and should not be relied on to fix the response that was just sent.

The planner-loop evaluator is a control-flow judge. It is not a plugin extractor. It evaluates the trajectory of tool calls and determines whether to stop, continue, or run a queued tool.
