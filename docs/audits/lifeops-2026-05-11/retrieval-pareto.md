## Notes

- Token contribution uses a per-action length proxy (`40 + nameLength/4`). Use this for relative ranking across K, not absolute prompt-token estimates.
- Pass-rate is the fraction of samples where every action the planner ultimately invoked appears inside the candidate top-K. Excludes samples without `selectedActions`.
- When no K meets the recall floor for a tier, the recommendation falls back to the K with the highest absolute recall.

## Measurement provenance (W6-G2, 2026-05-11)

This sweep is the first real run of the measurement pipeline against the
LifeOpsBench trajectory corpus. Source: 481 LifeOpsBench trajectories
(479 with valid `correctActions`) replayed through
`retrieveActions(measurementMode: true)` via
`scripts/lifeops-retrieval-replay.mjs`. The replay uses each trajectory's
embedded `BENCHMARK CONTEXT` tools manifest as the action catalog and
the actually-invoked tool (unwrapped from the `BENCHMARK_ACTION` shim) as
the `correctActions` ground truth.

### Stage interpretation

- BM25 carries the bulk of recall (~0.96 @ K=5) in this corpus. Most
  LifeOpsBench queries have explicit verb/object tokens that match
  action keyword text directly.
- `exact` + `regex` + `keyword` collectively recover all the same
  candidates earlier in the rank (155 + 197 + 22 first-appears) but
  cap out at ~0.39 alone ‚Äî they need the BM25 fusion to chase down
  the long-tail names.
- `embedding` and `contextMatch` show 0.00 recall here because the
  replay does not pass an `ActionEmbeddingTieBreaker` or
  `selectedContexts` (those flow from production runtime state the
  trajectory replayer does not reconstruct). In production both
  stages do fire; their absence here means the measured numbers are
  a **conservative lower bound** for the real retrieval funnel.

### Default-tier recommendation

Recall at K=5 (0.98) is effectively saturated ‚Äî K=8, 12, 20 each
add < 1pp recall for substantial token cost. The measured optimum
for the LifeOps slice is K=5 across all tiers. The pre-existing
heuristic defaults (small=5 / mid=8 / large=12 / frontier=20) are
**looser than the measured floor**, leaving prompt-tokens on the
table without buying additional planner recall on this corpus.

Apply the measured optimum to `mid` (8‚Üí6) and `large` (12‚Üí8) ‚Äî keep
some margin above the measured K=5 to absorb queries the replay
didn't cover (embedding/contextMatch stages on long-tail actions).
Hold `small=5` (already optimal) and `frontier=12` (we still want
the wider slate when context budget allows) to avoid over-fitting
to a 479-sample slice.
