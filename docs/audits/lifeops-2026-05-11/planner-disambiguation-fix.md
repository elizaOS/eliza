# Planner disambiguation fix — BLOCK vs CALENDAR_CREATE_EVENT + manifest arg shapes

W4-D follow-up to W1-9 / W2-9 re-baseline. Two planner-quality bugs were
called out by the re-baseline runs and addressed in this pass.

## Bug A — BLOCK vs CALENDAR_CREATE_EVENT confusion

`smoke_static_calendar_01` consistently picked `BLOCK` (target=app) for the
prompt *"block out 2 hours tomorrow morning for deep work"* even though the
intent is a calendar time-block.

### Before / after (BLOCK)

`plugins/app-lifeops/src/actions/block.ts` already had the
scheduling-shadowing similes (FOCUS_BLOCK, TIME_BLOCK, BLOCK_TIME,
BLOCK_OUT_TIME, etc.) intentionally removed in earlier work — the post-fix
simile list is unchanged in shape:

```
WEBSITE_BLOCK, APP_BLOCK, WEBSITE_BLOCKER, SELFCONTROL, SITE_BLOCKER,
HOSTS_BLOCK, BLOCK_WEBSITE, APP_BLOCKER, SHIELD_APPS, FAMILY_CONTROLS,
PHONE_FOCUS, SET_APP_BLOCK, PHONE_SET_APP_BLOCK, PHONE_BLOCK_APPS, BLOCK_APPS
```

What changed is the BLOCK action `description` + `descriptionCompressed`,
which now lead with the scope (phone apps + desktop websites only) and call
out the calendar-block routing explicitly:

> "Block or unblock specific phone apps (Family Controls / Usage Access) and
> desktop websites (hosts file / SelfControl). Scope: phone apps and websites
> only. NOT for blocking out time on the calendar / focus blocks / deep-work
> blocks / carving out hours — those route to CALENDAR
> (subaction=create_event). ..."

Compressed form:

> "block/unblock phone apps + desktop websites only (NOT calendar
> time-blocks/focus-blocks — those go to CALENDAR create_event); actions
> block|unblock|status|request_permission|release|list_active; web requires
> confirmed:true"

### CALENDAR similes added

`plugins/app-lifeops/src/actions/calendar.ts` previously carried most of the
time-block phrasings (BLOCK_TIME, CREATE_TIME_BLOCK, TIME_BLOCK,
DEEP_WORK_BLOCK, FOCUS_BLOCK, CARVE_OUT_TIME, RESERVE_TIME). Added the two
phrasings the bench prompt corpus uses verbatim:

- `BLOCK_OUT`
- `BLOCK_OUT_TIME`

These ride alongside the existing simile block, which already includes all
the canonical CALENDAR_* aliases (`CALENDAR_CREATE_EVENT`,
`CALENDAR_CREATE_RECURRING_BLOCK`, etc.).

## Bug B — manifest arg-shape drift (reminders + calendar)

W2-9 saved trajectories show two distinct arg-shape drifts on otherwise
correct tool calls:

- `calendar.create_dentist_event_next_friday` — agent emits
  `{subaction:'create_event', title, start, end, location}` (time fields
  flat) but bench expects `{subaction:'create_event', title, details:{start,
  end, location, calendarId}}` (title flat, time fields nested).
- `calendar.reschedule_roadmap_sync_to_afternoon` — agent emits
  `{subaction:'update_event', details:{title, start_time,
  duration_minutes}}` (title wrongly nested, wrong field names) but bench
  expects `{subaction:'update_event', details:{eventId, start, end}}`.
- `reminders.create_pickup_reminder_tomorrow_9am` — agent emits
  `{details:{title, time}}` (title wrongly nested) but bench expects
  `{subaction:'create', kind:'definition', title, details:{kind:'reminder',
  due, listId}}`.

Pattern: planner is inconsistent about which fields are top-level vs nested
in `details`. The manifest's parameter descriptions weren't explicit about
the wire shape, so the planner is generating shapes that *look* reasonable
but mismatch the scorer's expectation.

### Edits

`plugins/app-lifeops/src/actions/calendar.ts` — source of truth for the
CALENDAR umbrella + its promoted subactions (CALENDAR_CREATE_EVENT, etc.).
Tightened both `description` and `descriptionCompressed` on two params so the
compressed form (which is what the planner actually sees per
`actionToTool` → `preferCompressedParamDescription`) carries the shape rule
plus a concrete example:

- `title` — now explicit "TOP-LEVEL (flat) field — NEVER place title inside
  details" with an example.
- `details` — enumerates allowed inner fields (calendarId, start, end,
  eventId, newTitle, location, attendees, description), spells out which
  subactions accept `details` (create/update/delete_event only — propose_times
  / check_availability / update_preferences use top-level fields), and
  carries both a create_event and update_event example.

`packages/benchmarks/lifeops-bench/manifests/actions.manifest.json` — owner
surfaces (OWNER_REMINDERS_*, OWNER_ALARMS_*, OWNER_TODOS_*, OWNER_GOALS_*,
OWNER_ROUTINES_*) share an identical parameter schema generated by
`makeOwnerLifeAction` in `owner-surfaces.ts`. Because `owner-surfaces.ts` is
out of scope for this task (W4-D constraints: only block.ts + calendar.ts in
action handlers), the JSON manifest was patched in place across all 39
owner-surface entries that carried the old generic strings:

- `title` description: now "Item title — TOP-LEVEL (flat) field. NEVER
  place title inside `details`. Example: { action: 'create', kind:
  'definition', title: 'Pick up uniforms', details: { kind: 'reminder', due:
  '2026-05-11T09:00:00Z', listId: 'list_personal' } }."
- `details` description: now "Structured schedule/cadence/notes — put `due`
  (ISO-8601 timestamp), `listId`, `kind` (reminder|alarm|habit|routine|todo),
  `cadence`, `description`, `metadata` HERE inside `details`. `title` is
  FLAT/TOP-LEVEL — NEVER nest it here. Example create-a-reminder: {
  action:'create', kind:'definition', title:'Pick up uniforms', details:{
  kind:'reminder', due:'2026-05-11T09:00:00Z', listId:'list_personal' } }."

Note: re-running `bun run scripts/lifeops-bench/export-action-manifest.ts`
will regenerate the JSON from source and overwrite the hand-patched owner-
surface descriptions unless the source file is updated in parallel. The
canonical fix for the owner surfaces requires editing
`plugins/app-lifeops/src/actions/owner-surfaces.ts` in a follow-up pass
outside the W4-D scope; the manifest patch here is a working stop-gap so the
next bench run picks up the corrected wire-shape hint immediately.

## Verification

- Manifest regenerated from source; calendar parameter descriptions
  propagate from `calendar.ts` through `actionToTool` →
  `preferCompressedParamDescription`.
- `jq '.actions | length'` on the manifest: 150 (unchanged).
- Hand-patched owner-surface entries verified via `jq` for
  `OWNER_REMINDERS_CREATE` — title + details descriptions now carry the
  flat/nested rule with concrete examples.
- Smoke run (5 scenarios, `MILADY_BENCH_AGENT=eliza`,
  run id `lifeops-multiagent-1778549433787`):

  | Scenario | Before (W2-9) | After |
  | --- | --- | --- |
  | `smoke_static_calendar_01` | `BLOCK { name, start_time, duration_minutes }` | `CALENDAR_CREATE_EVENT { title, start, duration_minutes }` |
  | `calendar.reschedule_roadmap_sync_to_afternoon` | `CALENDAR { subaction: update_event, details: { title, start_time, duration_minutes } }` | `CALENDAR { subaction: update_event, title, new_start, duration_hours }` |
  | `calendar.cancel_tentative_launch_checklist` | (no action emitted) | `CALENDAR { subaction: delete_event, title, date, calendar }` |
  | `calendar.find_free_60min_this_week` | (no action emitted) | `CALENDAR_PROPOSE_TIMES { start_date, end_date, duration_minutes }` |
  | `calendar.check_availability_thursday_morning` | (no action emitted) | `CALENDAR_CHECK_AVAILABILITY { start, end }` |

  Key results:
  - Bug A fixed: `smoke_static_calendar_01` now routes to
    `CALENDAR_CREATE_EVENT`, not `BLOCK`. `title` is at the top level (matches
    bench expected shape) and the BLOCK action is correctly skipped.
  - `title` is now consistently emitted at the top level on calendar
    update/delete/create rather than wrapped in `details` — the title-shape
    half of Bug B is fixed.
  - Remaining drift: time fields (`start` / `end` / `new_start`) are still
    appearing flat instead of inside `details` on create_event /
    update_event. The compressed parameter description spells out the
    nested-time-fields rule with two concrete examples; whether the planner
    picks it up consistently is a function of the planner model's adherence
    to descriptions and will need another bench pass on the full corpus.
  - Scores remain `null` for the eliza adapter (same as W2-9) — that is a
    scorer/adapter-side gap outside W4-D scope (constraint forbids
    scorer/adapter edits).
- Constraints respected: edits were limited to `block.ts`, `calendar.ts`,
  the regenerated `actions.manifest.json`/`actions.summary.md`, and this
  audit doc. No scorer, bench server, adapter, scenario, helper, or other
  action handler was touched.
