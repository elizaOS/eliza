<div id="prerequisites">
## Prérequis
</div>

Avant d'installer Eliza, assurez-vous de disposer de l'environnement d'exécution requis :

| Prérequis | Version | Comment vérifier | Installer |
|-----------|---------|-----------------|-----------|
| **Node.js** | >= 22 | `node --version` | [nodejs.org](https://nodejs.org) ou `nvm install 22` |

Pour **compiler depuis les sources** ou pour le développement, vous aurez également besoin de :

| Prérequis | Version | Comment vérifier | Installer |
|-----------|---------|-----------------|-----------|
| **Bun** | latest | `bun --version` | `curl -fsSL https://bun.sh/install \| bash` |

<Info>
Node 22+ est requis car Eliza utilise les modules ES natifs, le préfixe de protocole `node:`, et d'autres API Node modernes. Bun n'est nécessaire que pour le développement et la compilation depuis les sources — il est utilisé comme gestionnaire de paquets et exécuteur de scripts. Le paquet npm publié et les scripts d'installation fonctionnent uniquement avec Node.js.
</Info>

Vous aurez également besoin d'une clé API d'au moins un fournisseur de modèles (Anthropic, OpenAI, OpenRouter, etc.) — sauf si vous prévoyez d'utiliser [Ollama](https://ollama.com) pour une inférence entièrement locale, qui ne nécessite aucune clé API.

<div id="desktop-app-recommended">
## Application de bureau (Recommandée)
</div>

L'application de bureau est le moyen le plus simple de commencer. Elle intègre l'ensemble du runtime Eliza, le tableau de bord web et la passerelle dans une application native Electrobun avec mises à jour automatiques et intégration dans la barre système.

Téléchargez depuis **[GitHub Releases](https://github.com/eliza-ai/eliza/releases/latest)** :

| Plateforme | Fichier | Notes |
|------------|---------|-------|
| macOS (Apple Silicon) | [`Eliza-arm64.dmg`](https://github.com/eliza-ai/eliza/releases/latest) | Macs M1/M2/M3/M4 |
| macOS (Intel) | [`Eliza-x64.dmg`](https://github.com/eliza-ai/eliza/releases/latest) | Macs d'avant 2020 |
| Windows | [`Eliza-Setup.exe`](https://github.com/eliza-ai/eliza/releases/latest) | Windows 10+ |
| Linux | [`Eliza.AppImage`](https://github.com/eliza-ai/eliza/releases/latest) / [`.deb`](https://github.com/eliza-ai/eliza/releases/latest) | AppImage ou paquet Debian |
| iOS | App Store (bientôt) | iPhone et iPad |
| Android | [Google Play](https://play.google.com/store/apps/details?id=ai.eliza.app) / [APK](https://github.com/eliza-ai/eliza/releases/latest) | Android 10+ |

Toutes les versions sont signées et notariées. Aucun avertissement Gatekeeper sur macOS.

<div id="verify-download-integrity">
### Vérifier l'intégrité du téléchargement
</div>

```bash
cd ~/Downloads
curl -fsSLO https://github.com/eliza-ai/eliza/releases/latest/download/SHA256SUMS.txt
shasum -a 256 --check --ignore-missing SHA256SUMS.txt
```

<div id="cli-install">
## Installation CLI
</div>

<Tabs>
  <Tab title="macOS / Linux">
    Le script d'installation télécharge la dernière version, place le binaire dans votre PATH et lance la configuration initiale :

    ```bash
    curl -fsSL https://get.eliza.ai | bash
    eliza setup
    ```

    Ensuite, lancez Eliza :

    ```bash
    eliza
    ```
  </Tab>
  <Tab title="Windows (PowerShell)">
    ```powershell
    irm https://get.eliza.ai/install.ps1 | iex
    ```

    Après l'installation, ouvrez une nouvelle fenêtre de terminal et exécutez :

    ```powershell
    eliza setup
    eliza
    ```
  </Tab>
  <Tab title="Bun">
    Si vous avez Bun installé :

    ```bash
    bun install -g elizaai
    eliza setup
    eliza
    ```
  </Tab>
  <Tab title="npm">
    Si vous préférez npm :

    ```bash
    npm install -g elizaai
    eliza setup
    eliza
    ```
  </Tab>
  <Tab title="Homebrew">
    ```bash
    brew tap eliza-ai/eliza
    brew install eliza          # CLI
    brew install --cask eliza   # Desktop app (macOS only)
    ```
  </Tab>
  <Tab title="Snap">
    ```bash
    sudo snap install eliza
    eliza setup
    ```

    Disponible sur Ubuntu, Fedora, Manjaro et toute distribution avec [snapd](https://snapcraft.io/docs/installing-snapd). Les paquets Snap se mettent à jour automatiquement en arrière-plan.

    Pour les dernières versions de développement :
    ```bash
    sudo snap install eliza --edge
    ```
  </Tab>
  <Tab title="Flatpak">
    ```bash
    flatpak install flathub ai.eliza.Eliza
    flatpak run ai.eliza.Eliza
    ```

    Ou installez en local depuis un [bundle de release](https://github.com/eliza-ai/eliza/releases/latest) :
    ```bash
    flatpak --user install eliza.flatpak
    ```
  </Tab>
  <Tab title="APT (Debian / Ubuntu)">
    ```bash
    # Add the repository
    curl -fsSL https://apt.eliza.ai/gpg.key | sudo gpg --dearmor -o /usr/share/keyrings/eliza.gpg
    echo "deb [signed-by=/usr/share/keyrings/eliza.gpg] https://apt.eliza.ai stable main" | \
      sudo tee /etc/apt/sources.list.d/eliza.list

    # Install
    sudo apt update && sudo apt install eliza
    ```

    Fonctionne sur Debian 12+, Ubuntu 22.04+, Linux Mint 22+, Pop!_OS et autres dérivés de Debian. Les mises à jour s'effectuent via `apt upgrade`.
  </Tab>
</Tabs>

<div id="build-from-source">
## Compiler depuis les sources
</div>

Pour les contributeurs ou toute personne souhaitant la dernière version de développement :

<Steps>
  <Step title="Cloner le dépôt">
    ```bash
    git clone https://github.com/eliza-ai/eliza.git
    cd eliza
    ```
  </Step>
  <Step title="Installer les dépendances">
    ```bash
    bun install
    ```

    Le hook `preinstall` initialise les sous-modules git en premier ; le `postinstall` qui suit s'assure que le répertoire des compétences existe, lie le serveur de navigateur et corrige les dépendances.
  </Step>
  <Step title="Tout compiler">
    ```bash
    bun run build
    ```

    Cela compile le code source TypeScript avec `tsdown`, écrit les informations de build et construit l'application web dans `apps/app`.

    <Info>
    `bun run build` exécute le build de production via Node (`scripts/run-production-build.mjs`).
    </Info>
  </Step>
  <Step title="Lancer Eliza">
    ```bash
    bun run start
    ```

    Cela lance la passerelle et le serveur API. Au premier lancement, vous serez guidé à travers le flux d'intégration interactif.
  </Step>
</Steps>

<div id="development-mode">
## Mode développement
</div>

Eliza dispose de plusieurs modes de développement selon ce sur quoi vous travaillez :

<CodeGroup>
```bash Full dev (runtime + UI hot reload)
bun run dev
```

```bash UI only (web dashboard hot reload)
bun run dev:ui
```

```bash Desktop app (Electrobun with live reload)
bun run dev:desktop
```

```bash Desktop app + Vite HMR
bun run dev:desktop:watch
```
</CodeGroup>

Le script `dev` utilise `scripts/dev-ui.mjs` pour démarrer le runtime avec le rechargement à chaud pour le backend et l'interface web. Le script `dev:desktop` lance l'application Electrobun avec le rechargement en direct.

<div id="environment-setup">
## Configuration de l'environnement
</div>

Eliza stocke tout son état dans `~/.eliza/` par défaut. Au premier lancement, le client
commence par choisir ou se connecter à un serveur. Si vous créez un serveur local, vous
pouvez toujours préconfigurer les paramètres d'amorçage et les clés de fournisseur avec
des variables d'environnement ou des fichiers de configuration.

<div id="api-keys">
### Clés API
</div>

Vous n'avez besoin de clés de fournisseur locales que lorsque le serveur local doit les
utiliser. Si vous prévoyez de vous connecter à un serveur LAN, distant ou Eliza Cloud
existant, ce serveur peut déjà gérer le routage et les identifiants du fournisseur. Pour
l'amorçage local, vous pouvez définir les clés de fournisseur dans
`~/.eliza/eliza.json` sous le champ `env` ou dans `~/.eliza/.env` :

<CodeGroup>
```bash Anthropic (recommended)
export ANTHROPIC_API_KEY="<ANTHROPIC_API_KEY>"
```

```bash OpenAI
export OPENAI_API_KEY="<OPENAI_API_KEY>"
```

```bash OpenRouter
export OPENROUTER_API_KEY="<OPENROUTER_API_KEY>"
```

```bash Google Gemini
export GOOGLE_API_KEY="<GOOGLE_API_KEY>"
```

```bash Ollama (local, no key needed)
export OLLAMA_BASE_URL="http://localhost:11434"
```
</CodeGroup>

<div id="ports">
### Ports
</div>

| Service | Port par défaut | Variable d'environnement |
|---------|----------------|--------------------------|
| Dashboard + API | `2138` | `ELIZA_PORT` |
| Passerelle (WebSocket) | `18789` | `ELIZA_GATEWAY_PORT` |
| Webhook WeChat | `18790` | `ELIZA_WECHAT_WEBHOOK_PORT` |
| Tableau de bord principal | `2142` | `ELIZA_HOME_PORT` |

En mode développement (`bun run dev`), l'API s'exécute sur un port séparé :

| Service | Port par défaut | Variable d'environnement |
|---------|----------------|--------------------------|
| API + WebSocket (dev) | `31337` | `ELIZA_API_PORT` |
| Interface Dashboard (dev) | `2138` | `ELIZA_PORT` |

```bash
# Example: custom ports
ELIZA_PORT=3000 ELIZA_GATEWAY_PORT=19000 eliza start
```

<div id="state-directory">
### Répertoire d'état
</div>

| Variable | Défaut | Description |
|----------|--------|-------------|
| `ELIZA_STATE_DIR` | `~/.eliza` | Répertoire racine pour tout l'état de Eliza |
| `ELIZA_CONFIG_PATH` | `~/.eliza/eliza.json` | Chemin explicite du fichier de configuration |

<div id="security-api-token">
## Sécurité : jeton API
</div>

Le serveur API se lie à `127.0.0.1` (boucle locale) par défaut — seule votre machine locale peut y accéder. Si vous exposez le serveur sur le réseau (par exemple, `ELIZA_API_BIND=0.0.0.0` pour des déploiements en conteneur ou dans le cloud), vous **devez** définir un jeton d'authentification :

```bash
echo "ELIZA_API_TOKEN=$(openssl rand -hex 32)" >> ~/.eliza/.env
```

<Warning>
Sans jeton sur une adresse de liaison publique, toute personne pouvant atteindre le serveur obtient un accès complet au tableau de bord, aux points de terminaison des agents et aux opérations de portefeuille.
</Warning>

<div id="verify-installation">
## Vérifier l'installation
</div>

Après l'installation, confirmez que tout fonctionne :

```bash
# Check version
eliza --version

# Show configured model providers
eliza models

# Bootstrap or validate local setup
eliza setup
```

La commande `models` liste les fournisseurs de modèles configurés et leur statut pour le
contexte du serveur local sélectionné. La commande `setup` amorce la configuration locale
si nécessaire et valide votre espace de travail.

<div id="troubleshooting">
## Dépannage
</div>

<div id="node-version-too-old">
### Version de Node trop ancienne
</div>

```
Error: Eliza requires Node.js >= 22
```

Mettez à jour Node.js : `nvm install 22 && nvm use 22`, ou téléchargez depuis [nodejs.org](https://nodejs.org).

<div id="bun-not-found">
### Bun introuvable
</div>

```
Error: bun: command not found
```

Installez Bun :
```bash
curl -fsSL https://bun.sh/install | bash
```

Puis redémarrez votre terminal ou exécutez `source ~/.bashrc` / `source ~/.zshrc`.

<div id="port-already-in-use">
### Port déjà utilisé
</div>

```
Error: EADDRINUSE :::2138
```

Un autre processus utilise le port par défaut. Arrêtez-le ou utilisez un port différent :
```bash
ELIZA_PORT=3000 eliza start
```

<div id="permission-denied-on-install-script">
### Permission refusée sur le script d'installation
</div>

```bash
chmod +x ~/.eliza/bin/eliza
```

<div id="pglite-database-corruption">
### PGLite / corruption de base de données
</div>

Si vous voyez des erreurs de base de données au démarrage, Eliza tentera une récupération automatique. Si cela échoue, vous pouvez réinitialiser la base de données :

```bash
rm -rf ~/.eliza/workspace/.eliza/.elizadb
eliza start
```

<Warning>
La suppression du répertoire de la base de données efface tout l'historique des conversations et la mémoire. Utilisez ceci en dernier recours.
</Warning>

<div id="macos-gatekeeper-blocks-the-app">
### macOS Gatekeeper bloque l'application
</div>

L'application de bureau est signée et notariée. Si Gatekeeper la bloque malgré tout :

```bash
xattr -cr /Applications/Eliza.app
```

<div id="local-macos-desktop-builds-unsigned-no-apple-certs">
### Builds locaux du bureau macOS (non signés / sans certificats Apple)
</div>

Les **releases** GitHub sont signées et notariées. Si vous **empaquetez l'application Electrobun localement** sans identifiants Developer ID, définissez **`ELECTROBUN_SKIP_CODESIGN=1`** (ou laissez les secrets non définis dans le CI) pour que le build n'attende pas la signature Apple.

Sur **macOS**, après `node scripts/desktop-build.mjs package` (depuis la racine du dépôt), le constructeur de bureau exécute **`apps/app/electrobun/scripts/local-adhoc-sign-macos.ts`** sur le **`.app`** produit pour que le bundle soit **signé ad hoc** pour les tests locaux. Vous pouvez aussi l'exécuter manuellement :

```bash
cd apps/app/electrobun && bun scripts/local-adhoc-sign-macos.ts /path/to/Eliza.app
```

La signature ad hoc n'est **pas** une notarisation ; Gatekeeper peut encore nécessiter **Clic droit → Ouvrir** la première fois. Contexte complet du pipeline : [Build and release](/fr/build-and-release).
