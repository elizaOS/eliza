# Gemma 4 cutover — next-wave execution plans


## M6 — Kernel/quant strategy for Gemma 4 (read-only execution plan)

DECISIONS (all evidence-backed from the kernel sources, reference header, quant scripts, model_registry, and build/verify gates):

(1) KEEP for Gemma — weight-quant only. The KEEP set is the GGUF weight-quant path: `gguf_eliza1_apply.py` (`Q4_POLAR`/`q8_0`/f16 body + `--drop-mtp` for separate-drafter Gemma) plus the k-quant ladder (`gguf-q3_k_m/q4_k_m/q5_k_m/q6_k/q8_0_apply.py`) for the OmniVoice/ASR sidecars. CONFIRMED geometry-agnostic for dense Gemma weights: weight quant operates on `(out_features,in_features)` matmul tensors, independent of attention head geometry; `fused_turboquant_apply.py:64` already lists `gemma` in `_KNOWN_GOOD_ARCH_SUBSTRINGS` and `check_model_compatibility` accepts dense MQA. CAVEAT on the name "TurboQuant turbo3/turbo4/tcq": those named kernels (`reference/turbo_kernels.h`, `vulkan/turbo3*.comp`, `metal/turbo3*.metal`) are KV-CACHE compressors hard-wired to 128-element FWHT groups (`ELIZA_QK_TURBO3_GROUP 128`, `eliza_turbo_rotate_forward(float x[128])`, `block_turbo3_tcq` QK=128) — they are NOT weight-quant and are head_dim-coupled. So "TurboQuant weight-quant" in the plan == the GGUF body quant (`weight_quant` block in `gguf_eliza1_apply.py:459`), which is geometry-agnostic and stays REQUIRED. The turbo3/turbo4/turbo3_tcq KV kernels DROP for Gemma (see #2).

(2) DROP/disable for Gemma KV — QJL (K-cache) + PolarQuant (V-cache) + turbo3/turbo4/turbo3_tcq (KV). All three are head_dim-coupled: QJL builds per-(head_dim=128, proj_dim=256) JL matrices (`qjl_apply.py:235-265`), PolarQuant uses `block_size=128` WHT blocks (`polarquant_apply.py:64`), turbo* kernels are 128-group FWHT. Gemma's KV is MQA (n_head_kv=1) + windowed-SWA (512/1024) + shared-KV + dual head dims (512 global / 256 SWA) — already minimal, and the kernels' 128-geometry does not match either head dim. DECISION: Gemma KV uses STOCK q8_0 (no eliza KV kernel), driven by `swa_full=false` + bounded ctx-checkpoints + mmap-on PLE-pinned-to-CPU (M2 runtime defaults). The kernels stay COMPILED into libelizainference (legacy Qwen tiers + the build-symbol gate still need them; the omnivoice/ASR FFI symbols share the same build), but the Gemma manifest moves qjl/polarquant/turbo3_tcq from `kernels.required` → `kernels.optional`, and a Gemma-aware verify required-set replaces the legacy 8/8.

(3) Dead Qwen-hybrid code — SAFE TO DELETE vs RETARGET (full list in steps). DELETE: `inference/hybrid_cache.py` + `inference/test_hybrid_cache.py` (ElizaHybridCache only fires on `layer_types` containing `linear_attention` — never on dense Gemma; already flagged dead in `serve_local.py:109-115`); the entire `scripts/eagle3/` dir + `test_eagle3_pipeline.py` (same-model EAGLE3 drafter distillation, replaced by Gemma official separate drafters per `model_registry.py:286-290`); `serve_vllm.py` qwen3_next_mtp/`--language-model-only`/MoE-expert-parallel branches (Qwen3.5/3.6 only). RETARGET (do NOT delete — runtime-consumed): the `eagle3` manifest field in `plugins/.../manifest/schema.ts:320` + `validator.ts:528` is a generic spec-decode acceptance/speedup eval — rename to `mtp`/`specDecode`, keep the field; `serve_local.py` keep the script, delete only the `is_hybrid` branch; `_common.py` `full_attention_layer_indices`/`layer_types` helpers become dead once hybrid_cache + fused_turboquant hybrid-guard go (the fused_turboquant_apply.py hybrid EXIT_INCOMPATIBLE_ARCH guard at :137-162 can be dropped since Gemma is dense). WHAT BREAKS IF DELETED BLINDLY: deleting the `eagle3` manifest schema field breaks `validator.ts` + every published manifest that carries spec-decode evals → retarget instead; deleting `_common.head_dim_of` breaks turboquant/qjl/polar apply imports → keep it (still used by KV scripts even if KV is Gemma-optional).

(4) Per-platform max-opt for Gemma (MQA + SWA + shared-KV): CPU — stock q8_0 KV + NEON/AVX2 gemm, mmap PLE on disk, `swa_full=false`. CUDA — FlashAttention-2 over SWA windows + global, PLE pinned to host (not VRAM), q8_0 KV; no eliza KV kernel dispatch. Vulkan-Mali/Adreno — coopmat where present, PLE mmap (Vulkan can't run mmap=0 per #18317 — never force --no-mmap), q8_0 KV, windowed-SWA sizing; LiteRT-LM NPU backend (M4) is the real Mali/Tensor/QNN max-opt path (dense graphs are NPU-delegate-friendly — the core reason for the cutover). Metal — FA over SWA, PLE on unified memory, q8_0 KV; CoreML/MLX (M5) as the Apple in-process max-opt. LiteRT-NPU note: ships pre-converted Gemma `.litertlm` bundles doing PLE-mmap + windowed-SWA + NPU natively — do not reimplement; gate behind `-DELIZA_ENABLE_LITERT`.

## M10b — eliza-1 memory max-optimization (#8809): revive the dead arbiter LRU, replace static minRam-floor gating with dynamic fit-to-RAM quant+context selection, and add a desktop multi-modality memory-bench harness.

ROOT CAUSE of the dead LRU: `MemoryArbiter.evictToFit`/`residentFootprintMb` only sum `estimatedMb` over the arbiter's own `this.resident` map. Only `vision-describe` (600) and `image-gen` (1100) are ever registered as arbiter capabilities (service.ts:514/569). The two dominant resident consumers — TEXT and EMBEDDING — never go through the arbiter at all: text is owned by `LocalInferenceEngine`, which registers a `text-target` evictable role on its OWN `SharedResourceRegistry` (engine.ts:407, with a real `estimateTextResidentMb` at engine.ts:218 = weights+overhead), and embedding is loaded by a separate fused handle in the runtime handler (ensure-local-inference-handler.ts ~647) with no evictable role and no estimate. So `residentFootprintMb()` is near-zero in practice (it sees at most vision+imagegen), the `budgetMb()` is real (service.ts:498, totalmem−headroom), and `residentFootprint + incomingMb > budget` essentially never trips → the proactive fit path is dead. NOTE: `estimatedMb` is NOT literally hardcoded 0 for vision/imagegen; the "0 for text/embedding" is because text/embedding are absent from the arbiter's resident set, which is the real defect. The fix is to make the arbiter aware of the text-target and embedding footprints (either register them as arbiter capabilities, or have `residentFootprintMb`/`evictToFit` consult the engine's `SharedResourceRegistry` footprint), so the LRU operates over the true resident set.

DYNAMIC FIT gap: CLOSED 2026-06-23. `selectBestEliza1Fit` (device-fit.ts) already picked tier+context from free RAM for advisory surfaces. The live load path now wires the same policy shape through `context-fit.ts` + `active-model.ts`: when no explicit `contextSize` override is set, load args compute `maxFittingContext = floor((usableMb − weightMb − workingSet) / kvBytesPerToken)` with `estimateQuantizedKvBytesPerToken`, clamp to the bundle/catalog native ceiling, and then apply the mobile ceiling. Quant remains fixed stock q8_0 by Gemma geometry; `preferAccurateKvWhenHeadroom` remains deferred.

BENCH gap: desktop `memory-benchmark.ts` is text-only (loads each installed eliza-1 bundle, decode, RSS delta). The iOS path is `mobile_peak_rss_harness.mjs` (a needs-device stub) + `eliza1_memory_inventory.mjs` (disk-footprint inventory). No harness exercises embedding/vision/VAD residency or co-residency on desktop.
