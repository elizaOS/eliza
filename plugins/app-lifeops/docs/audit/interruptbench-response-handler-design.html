Response Handler — Field Evaluator Registration Pattern

Single Stage-1 LLM call. Modular fields. Abort intent is just another thread op.
Design doc v2 — flat schema, all required Companion to interruptbench-wave0-contract.md

v2 — flat schema correction

The original sketch in this doc nested fields under plan.* and extract.* with many optionals. After review, the actual schema is dramatically simpler: 

export const HANDLE_RESPONSE_SCHEMA: JSONSchema = {
  type: "object",
  additionalProperties: false,
  properties: {
    shouldRespond:        { type: "string", enum: ["RESPOND", "IGNORE"] },
    contexts:             { type: "array", items: { type: "string" } },  // includes "simple"
    intents:              { type: "array", items: { type: "string" } },
    candidateActionNames: { type: "array", items: { type: "string" } },
    replyText:            { type: "string" },
    facts:                { type: "array", items: { type: "string" } },
    relationships:        { type: "array", items: { type: "object", properties: {...}, required: ["subject","predicate","object"] }},
    addressedTo:          { type: "array", items: { type: "string" } },
    threadOps:            { type: "array", items: { /* registered by app-lifeops; abort lives here */ }},
  },
  required: [
    "shouldRespond", "contexts", "intents", "candidateActionNames",
    "replyText", "facts", "relationships", "addressedTo", "threadOps",
  ],  // ALL required. Empty array / empty string for N/A. Strict mode.
};

Changes from v1: flat (no plan.* / extract.* wrappers) · all required (no optionals — empty value for N/A) · simple is a context name, not a flag (contexts: ["simple"]) · no STOP (was indistinguishable from IGNORE) · no thought (was prompt theater) · no requiresTool / simple booleans (derivable from other fields) · new intents field (short verb phrases for routing). Each registered ResponseHandlerFieldEvaluator owns one of these fields, declares its description + schema-with-parameter-descriptions, and provides one handler that runs in pipeline order.

1. Verdict

The current response handler already runs one LLM call at message.ts:4222-4232 with a static HANDLE_RESPONSE tool schema. The schema is intentionally byte-stable for prompt-cache reasons (the comment at to-tool.ts:14-27 says so explicitly).

What's missing is per-field modularity. Today, every field of the response is either: hard-coded into HANDLE_RESPONSE_SCHEMA (to-tool.ts:71-123), or hard-coded as a keyword-driven "repair" chain (message.ts:2014-2336), or wired into a post-parse ResponseHandlerEvaluator patcher (response-handler-evaluators.ts).

The fix is to mirror the existing post-turn Evaluator pattern — which already does exactly this for the post-turn step (schema fragment + prompt fragment + parse + processors, composed into one LLM call). Apply the same shape to the response handler step. Abort intent then drops in naturally as one operation type in a threadOps field contributed by the work-thread evaluator. No regex. No separate call.

Net effect: Stage 1 becomes a registry of ResponseHandlerFieldEvaluator contributors. Each contributes a schema slice + a prompt slice + a parser + processors. The runtime composes them into one cache-stable LLM call. Hardcoded repair logic dies.

Problem. Modes (a) and (b) are inflexible. The schema is hardcoded; new fields require core edits. Repairs (b) are AI-slop keyword chains that ought to be domain plugins. Mode (c) post-parse can only react to what the LLM already produced — it can't ask the LLM for new structured data. So if a plugin wants the LLM to extract "thread operations to perform on the user's behalf" in the same call, there's no way to do that today.

The existing post-turn Evaluator pattern — already correct shape

From types/evaluator.ts:44-58:

export interface Evaluator<TOutput = JsonValue, TPrepared = unknown> {
  name: string;
  description: string;
  similes?: string[];
  priority?: number;
  providers?: string[];
  schema: JSONSchema;
  modelType?: ModelTypeName;

  shouldRun(context: EvaluatorRunContext): Promise<boolean>;
  prepare?(context: EvaluatorRunContext & { state: State }): Promise<TPrepared>;
  prompt(context: EvaluatorPromptContext<TPrepared>): string;
  parse?(output: unknown): TOutput | null;
  processors?: Array<EvaluatorProcessor<TOutput, TPrepared>>;
}

Critically, at services/evaluator.ts:81-89 the runtime merges all active evaluators into ONE JSON schema:

function buildMergedSchema(active: PreparedEntry[]): JSONSchema {
  return {
    type: "object",
    properties: Object.fromEntries(
      active.map(({ evaluator }) => [evaluator.name, evaluator.schema]),
    ),
    required: active.map(({ evaluator }) => evaluator.name),
    additionalProperties: false,
  };
}

One LLM call. N contributors. Each parses its own slice. Each runs its own processors. This is what we want for the response handler.

3. Proposed architecture

Replace the static HANDLE_RESPONSE_SCHEMA with a composed schema built from registered ResponseHandlerFieldEvaluators. Each field — including all the existing ones — becomes an evaluator. The runtime composes their schemas into ONE call.

Key properties

PropertyBehavior
One LLM call Stage 1 is still a single useModel(RESPONSE_HANDLER, ...) call. We don't multiply the cost.
Cache-stable schema Registered field set is fixed at plugin-load time. Same set every turn → same schema bytes → cache hits.
Conditional activation, not conditional schema shouldRun doesn't add/remove fields from the schema; it adds/removes natural-language instructions in the system prompt telling the LLM when to populate the field. The field stays declared (and nullable) in the schema. Cache stays warm. (Pattern lifted from BAML + Anthropic cache hierarchy.)
Per-field validation + handler Each field has its own parse (validate/transform the LLM's value for this slice) and its own processors[] (run side effects: thread mutations, abort calls, retrieval kicks, etc.).
Error isolation One field's parse failing or processor throwing does not nuke the run. Soft-fail (BAML @check) attaches an error to that field; hard-fail (BAML @assert) drops the field and optionally surfaces a targeted re-ask.
Determinism for benchmarks Registration order is sorted by priority then name. Schema is byte-stable across benchmark runs with the same plugin set. InterruptBench's scripted-LLM fixtures stay valid.

4. ResponseHandlerFieldEvaluator contract

The full proposed interface, mirroring Evaluator closely:

export interface ResponseHandlerFieldEvaluator<TValue = unknown, TPrepared = unknown> {
  // Identity. `name` becomes the field's JSON property name in plan.{name}.
  name: string;
  description: string;
  priority?: number; // default 100. Lower runs first.

  // === Schema contribution (cache-relevant) ===
  // This is the JSON schema for the LLM's value for this field. Always nullable.
  schema: JSONSchema;
  // Whether to include this field at all in the composed schema. Static — evaluated
  // once at registration. Use shouldRun for per-turn activation.
  alwaysDeclare?: boolean; // default true. Set false only for cross-tenant fields.

  // === Per-turn activation (does NOT change schema) ===
  // Cheap synchronous-ish gate. When false, the system prompt omits this field's
  // prompt slice, and the LLM is implicitly told not to populate it. Schema stays
  // identical across turns.
  shouldRun(ctx: ResponseHandlerFieldContext): boolean | Promise<boolean>;

  // Optional: gather state for the prompt slice and processors.
  prepare?(ctx: ResponseHandlerFieldContext): Promise<TPrepared>;

  // === Prompt contribution ===
  // Returns the natural-language instructions for the LLM about THIS field.
  // Goes into the system prompt's stable cacheable region.
  prompt(ctx: ResponseHandlerFieldContext & { prepared?: TPrepared }): string;

  // === Per-field parsing ===
  // Validate / transform the LLM's value for this field. Return null to soft-fail.
  // Throw to hard-fail (drops field, surfaces error).
  parse?(value: unknown, ctx: ResponseHandlerFieldContext): TValue | null;

  // === Side-effect processors ===
  processors?: Array<ResponseHandlerFieldProcessor<TValue, TPrepared>>;
}

export interface ResponseHandlerFieldProcessor<TValue, TPrepared> {
  name?: string;
  priority?: number;
  process(ctx: {
    runtime: IAgentRuntime;
    message: Memory;
    state: State;
    prepared?: TPrepared;
    value: TValue;
    messageHandler: MessageHandlerResult; // MUTABLE — patch siblings if needed
    abortSignal: AbortSignal; // turn-scoped controller
  }): Promise<void>;
}

export interface ResponseHandlerFieldContext {
  runtime: IAgentRuntime;
  message: Memory;
  state: State;
  senderRole: SenderRole;
  availableContexts: readonly ContextDefinition[];
}

Prompt slice (only emitted when shouldRun is true)
### Thread operations
You can perform operations on work threads. Active threads visible to you:
- wt-7faaa3 ("draft the offsite plan"): waiting on confirmation since 12 min ago
- wt-9b21c0 ("research espresso machines"): running, last update 5s ago

When the user wants to:
- start a new long-running task → emit { type: "create", instruction: "..." }
- correct or refocus an ongoing thread → emit { type: "steer", workThreadId, instruction }
- cancel / stop / abort what the agent is currently doing → emit { type: "abort", workThreadId? }
- combine two threads → emit { type: "merge", workThreadId: TARGET, sourceWorkThreadIds: [...] }

If the user expresses no thread intent, omit threadOps entirely (or emit []).
parse

Validate that workThreadId references a real thread the user owns or can mutate; validate that merge sources are all in the current channel; validate that abort only refers to the currently-active turn or an active thread.

processors
  1. Abort-first processor. If any op is { type: "abort" }, call runtime.abortTurn(roomId, reason) immediately. This signals all in-flight tool calls, sub-agents, and the planner via the shared AbortController. Cleanup window kicks in (5s) for WIP commits.
  2. Thread-control processor. Translate remaining ops to lifeops_thread_control action invocations. Reuses the existing work-thread store, transaction wrapping, optimistic concurrency.
  3. Reply-suppression processor. If abort was performed, suppress the planner route and emit a short ack reply ("stopped — partial work preserved").
How abort intent comes through naturally

User says "stop" / "nvm" / "actually wait don't do that". The LLM has the thread context in its prompt (because shouldRun was true). It emits { threadOps: [{ type: "abort", reason: "user retracted" }] } as part of the same Stage-1 call it was already making. No keyword regex. No separate intent classifier. No extra LLM call.

The migration

Each of the 7 keyword-driven repair functions at message.ts:2014-2336 becomes a field evaluator owned by its respective plugin. The schema-fragment + prompt-instruction approach gives the LLM the same routing power without keyword matching.

Example: approval resolution. Today, if the LLM doesn't return contexts and the message contains "approve" or "reject", message.ts:2063 injects a fallback plan adding the approvals context. With field evaluators, the approvals plugin contributes:

6. Schema composition & caching strategy

From the cache research: the existing response handler already does the right thing. Tool schema is byte-stable, system prompt is built from stable: true segments, providers cache per-turn or per-room. The field evaluator pattern preserves all of this — provided we follow a few rules.

Rule 1: schema is byte-stable across turns of the same boot

All registered field evaluators with alwaysDeclare !== false contribute their schema to HANDLE_RESPONSE_SCHEMA at runtime startup. The composed schema is computed once and reused for every Stage-1 call. Plugin load is the only time the schema changes.

Why this works. Anthropic's cache invalidation rule is that any change to tool definitions invalidates the tools-and-everything-after cache region. By making the schema deterministic (sorted by name, computed once), every Stage-1 call shares the same tools-block bytes. Anthropic gives us cache hits, OpenAI gives us extended-retention prompt cache key matches, Cerebras gets a stable prompt cache key, local Qwen/Ollama get KV slot pinning by conversationId.

Rule 2: shouldRun affects the prompt, not the schema

Per-turn variability lives in the natural-language prompt instructions, not the schema. When shouldRun returns false, the evaluator's prompt() output is omitted from the system prompt. The schema still lists the field — but the LLM has no instruction to populate it, and any non-null output is ignored by the post-parse step.

Because Anthropic places its cache breakpoints at segment boundaries, this means: each evaluator's prompt slice is its own segment, marked stable: true if the evaluator's prompt() output depends only on stable data (rare), or non-stable if it depends on per-turn data like the list of active threads (common).

Rule 3: dynamic data flows through providers, not through schema

"Active threads in this room right now" is dynamic. It goes into a Provider's output (existing pattern at plugins/app-lifeops/src/providers/work-threads.ts) which is rendered into a dynamic prompt block. The threadOps field evaluator's prompt() reads from this rendered text — but it does not bake the thread list into its prompt slice. Net: cache stays warm, the LLM still sees fresh data.

What invalidates the cache

  • Registering a new plugin that adds a field evaluator (one-time per boot)
  • Removing or re-ordering field evaluators (rare; only on hot-reload)
  • Changing the contexts enum (when context plugins load)
  • Changing a character's name/identity in the system prompt

None of these happen mid-conversation, so steady-state cache hit ratio is dominated by Rules 1–3.
