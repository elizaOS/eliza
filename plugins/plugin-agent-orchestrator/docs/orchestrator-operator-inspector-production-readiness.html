What We Have Now

Inspectable task room

The operator workbench can inspect task detail, sub-agent sessions, user and agent turns, reasoning blocks, system events, tool calls, artifacts, and usage. Inspector drawers preserve ids so polling refreshes can keep focus on the same logical record.

Normalized timeline

The backend now exposes a single mixed timeline endpoint for messages and events. The UI uses that endpoint for initial fetches, refreshes, and loading older history, which keeps tool, reasoning, notice, and message ordering consistent.

Recovery controls

Retry turn, rerun from event, restart, and restart with edited plan now exist as backend routes and UI client methods. The workbench exposes compact contextual recovery buttons from task, session, and timeline detail surfaces, plus a compact inspector plan editor that restarts a task from edited plan JSON. Operator input conflicts return explicit conflict errors instead of pretending to succeed.

Immutable plan revisions

Plans can be stored as durable revision records, listed, referenced by recovery actions, and applied to task state. Older persisted task documents load with an empty revision list for backwards compatibility.

Tool fidelity

ACP terminal tool updates are emitted into the timeline so the inspector keeps final status, raw output, structured output, and execute exit codes instead of leaving tools stuck at in-progress.

Side-effect honesty

User-message delivery and stop controls now report ACP-unavailable failures and mark affected sessions as failed, rather than silently recording a local success with no remote side effect.

Backend Surface

The durable orchestrator API now supports task lifecycle, room messaging, telemetry, timeline inspection, plan revisioning, and recovery.

Vision

Product direction

  1. Keep the primary experience as a live agent room, not a static workflow designer.
  2. Make every meaningful turn, tool call, session, artifact, decision, and recovery action inspectable.
  3. Let operators intervene with bounded, explicit side effects: message, pause, stop, retry, rerun, restart, fork, validate, and edit plan.
  4. Preserve the durable context that makes recovery auditable: ids, timestamps, raw tool records, source event ids, plan revisions, and validation evidence.

What we should avoid

  • Do not turn Eliza into a DAG editor.
  • Do not fabricate per-tool token or cost numbers.
  • Do not hide failed side effects behind optimistic UI success.
  • Do not discard timeline history needed for inspection.
  • Do not add destructive rerun behavior until there is a dedicated product and safety design.

Remaining Product Work

Richer recovery UX

Compact retry, rerun, and restart buttons now exist. The next UX step is a richer guided recovery flow with operator-authored instructions, plan selection, previews, and clearer before/after impact.

Guided plan editor UX

A minimal inspector JSON editor can restart from an edited plan and records the base revision plus edit summary. The next UX step is a guided editor with structured step controls, plan selection, previews, and clearer before/after impact.

Remote/mobile live updates

The workbench still relies on polling. A production remote/mobile version should use SSE or WebSocket updates and a registered tunnel service instead of repeated polling.

Provider and verification UX

Provider policy is inspectable but not actively changeable, and validation evidence is present but not yet a dedicated checklist surface.
