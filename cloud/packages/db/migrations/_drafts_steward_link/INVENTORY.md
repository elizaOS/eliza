# Privy → Steward identity column inventory

Generated by reading `packages/db/schemas/` and `packages/db/migrations/`.
DO NOT execute any migration based on this inventory without operator review.

## Summary

There are exactly **two** tables that hold a per-user Privy identifier
(`did:privy:...` shape) that needs to be linked to a Steward user ID:

1. `users.privy_user_id`
2. `user_identities.privy_user_id`

Both already have a parallel `steward_user_id` column added by the applied
migration `0061_add_steward_user_identity_columns.sql`. **Phase 1 of the plan
described in `cloud/AUTH_MIGRATION_NOTES.md` is therefore already in place
and applied.** The drafts in this directory cover Phase 3 (finalize), plus a
documentation-only Phase 2 entry describing the application-level backfill
that must run between them.

A third table — `agent_server_wallets` — has a `privy_wallet_id` column.
This is **not** a per-user identity; it is the Privy-side wallet ID for an
agent server wallet. It is already being migrated by the dual-provider
mechanism in applied migration `0058_add_steward_wallet_provider.sql`
(`wallet_provider`/`steward_agent_id` columns + check constraint). It does
**not** belong to the Steward-user-link plan; it follows its own
agent-wallet provider migration. See "Out of scope — `agent_server_wallets`"
below.

There are **zero** other tables that store a Privy user identifier. Every
other user-owned table references `users.id` (a uuid), which is unaffected
by this migration.

Source: `packages/db/schemas/agent-server-wallets.ts:31-39`.

This column does **not** participate in the Steward-user-link migration.
Its migration story is:
- Already-applied `0058_add_steward_wallet_provider.sql` introduced the
  dual-provider routing.
- New rows write `wallet_provider='steward'` + `steward_agent_id=<id>`.
- Existing Privy wallets stay on the `privy` provider until backfilled or
  rotated by a future agent-wallet operator runbook.

The Phase 3 draft in this directory does **not** drop `privy_wallet_id` or
relax `wallet_provider_id_check`. That is owned by the agent-wallet
migration, not this one.

Privy tenant ID on `organizations` to migrate from, so no draft is needed.

## Exceptions and edge cases

1. **Composite-key tables**: none. Every user-owned row uses `users.id`
   as its FK target, so there is no composite key involving Privy IDs to
   handle specially.

2. **`user_identities` rows with NULL `privy_user_id` after migration
   `0050`**: migration `0050_repair_existing_user_identity_privy_claims.sql`
   nulls out `user_identities.privy_user_id` for rows where the same
   Privy ID is claimed by multiple identity rows (deliberately, to
   require manual cleanup). For these rows, the only path back to a
   Steward link is via the matching `users.privy_user_id` (which is
   still set on the parent row). Phase 2 backfill must therefore source
   the link from `users.privy_user_id`, **not** from
   `user_identities.privy_user_id`. The drafted Phase 3 finalize
   migration accounts for this by checking `users.steward_user_id IS NOT
   NULL` rather than the projection row.

3. **Anonymous users**: `is_anonymous = TRUE` rows have `privy_user_id =
   NULL` and will remain `steward_user_id = NULL` until the user
   migrates the anonymous session into an authenticated session. These
   rows should NOT be flagged by the unlinked-row count query in
   pre-flight; the query in README.md filters them out.

4. **Eliza-app-only users** (Telegram/Discord/WhatsApp/phone-only,
   never logged into Privy): also have `privy_user_id = NULL`. Same
   exclusion applies.
